## Summary

Extract transcription from the ingest subsystem into a shared service domain, following the existing image-generation architecture. The result should provide one consistent transcription capability surface for:
- vault ingest
- audio editor transcript generation
- future transcript repair/alignment workflows

This plan also phases in additional providers while the refactor is underway:
1. keep and harden `groq` and `openai`
2. add native `mistral` transcription
3. add `deepgram`
4. add `assemblyai`

This is intentionally separate from `docs/plans/audio-editor-leaf-view-plan.md`. That plan depends on a stronger transcription foundation, but this work can be executed independently.

The new target shape should mirror the image stack:
- `ImageGenerationService` → `TranscriptionService`
- `ImageTypes` → `VoiceTypes`
- `BaseImageAdapter` → `BaseTranscriptionAdapter`
- provider-specific image adapters → provider-specific transcription adapters
- shared defaults/capability discovery instead of ingest-owned model branching

## What The Code Does Today

### Current Ownership Boundary
- Audio transcription is currently implemented inside ingest-specific files:
  - `src/agents/ingestManager/tools/services/IngestModelCatalog.ts`
  - `src/agents/ingestManager/tools/services/IngestCapabilityService.ts`
  - `src/agents/ingestManager/tools/services/TranscriptionService.ts`
- `DefaultsTab` resolves `Default transcription` from ingest capability discovery.
- `VaultIngestionManager` reads `defaultTranscriptionModel` and forwards provider/model into the ingestion pipeline.

### Current Strengths
- The repo already has a working transcription pipeline for:
  - `openai`
  - `groq`
  - `google`
  - `openrouter`
- Chunking already exists.
- There is already one global default transcription setting.

### Current Problems
- Transcription is owned by ingest instead of a shared service domain.
- Provider/model declarations are mixed into ingest concerns.
- The current return shape is not strong enough for editor use:
  - segment timing exists
  - word timing is not modeled as a first-class shared type
  - transcript prompts/glossaries are not modeled cleanly
- Provider expansion is awkward because every new transcription provider currently means more ingest branching.

### Existing Pattern To Copy
- Image generation already has the architecture we want:
  - `src/services/llm/ImageGenerationService.ts`
  - `src/services/llm/types/ImageTypes.ts`
  - `src/services/llm/adapters/BaseImageAdapter.ts`
  - provider-specific image adapters
- Provider settings/defaults are already handled centrally in:
  - `src/types/llm/ProviderTypes.ts`
  - `src/settings/tabs/DefaultsTab.ts`
  - `src/settings/tabs/ProvidersTab.ts`
  - `src/services/llm/validation/ValidationService.ts`

## Design Rules

### 1. Transcription becomes a first-class service
Do not keep transcription as an ingest-private concern.

Reason:
- multiple product surfaces will need it
- provider expansion becomes cleaner
- defaults and capability discovery become reusable

### 2. Ingest becomes a consumer
The ingest pipeline should call the shared transcription service. It should not remain the owner of provider branching logic.

### 3. Keep one global default transcription
Do not add a second top-level default setting for the audio editor.

Keep:
- `defaultTranscriptionModel: { provider, model }`

Refactor:
- the backing capability source
- the orchestration layer used by ingest/audio editor

### 4. Model the result shape for editor use
The shared result type should support:
- transcript text
- segments
- optional words
- optional speaker/utterance metadata
- optional provider-side confidence/diagnostics

### 5. Vendor rollout should be ingest-first
For `deepgram` and `assemblyai`, add them first as transcription providers, not full chat providers.

Reason:
- lower scope
- clearer product justification
- avoids unnecessary adapter work outside transcription

## Proposed Architecture

### Service Surface

```ts
interface TranscriptionRequest {
  audioData: ArrayBuffer;
  mimeType: string;
  fileName: string;
  provider?: string;
  model?: string;
  prompt?: string;
  glossary?: string[];
  requestWordTimestamps?: boolean;
  requestSpeakerLabels?: boolean;
}

interface TranscriptionWord {
  text: string;
  startSeconds: number;
  endSeconds: number;
  confidence?: number;
  speaker?: string;
}

interface TranscriptionSegment {
  startSeconds: number;
  endSeconds: number;
  text: string;
  confidence?: number;
  speaker?: string;
  words?: TranscriptionWord[];
}

interface TranscriptionResult {
  provider: string;
  model: string;
  text: string;
  durationSeconds?: number;
  segments: TranscriptionSegment[];
  metadata?: Record<string, unknown>;
}
```

### New Core Files
- `src/services/llm/TranscriptionService.ts`
- `src/services/llm/types/VoiceTypes.ts`
- `src/services/llm/adapters/BaseTranscriptionAdapter.ts`
- `src/services/llm/adapters/openai/OpenAITranscriptionAdapter.ts`
- `src/services/llm/adapters/groq/GroqTranscriptionAdapter.ts`
- `src/services/llm/adapters/mistral/MistralTranscriptionAdapter.ts`
- later:
  - `src/services/llm/adapters/deepgram/DeepgramTranscriptionAdapter.ts`
  - `src/services/llm/adapters/assemblyai/AssemblyAITranscriptionAdapter.ts`

### Capability Discovery

Introduce a shared transcription model catalog/service, analogous to image generation:
- shared provider/model declarations
- provider-specific availability checks
- `getModelsForProvider()`
- `getAvailableProviders()`

This can either live inside the new transcription service or in a sibling model-catalog module. The critical requirement is that it is no longer ingest-owned.

### Defaults Integration

`DefaultsTab` should continue rendering:
- `Default transcription`

But the options should come from the shared transcription capability layer, not from `IngestCapabilityService`.

### Compatibility Strategy

Short term:
- keep the current ingest `TranscriptionService.ts` as a shim
- route its implementation into the shared service

Long term:
- remove or collapse the ingest-local service once all call sites are migrated

## Provider Research And Rollout

### Phase A: Existing Providers

#### `groq`
- Keep as the default timestamp-capable path.
- Best fit for word-timestamp-first editor work in the current stack.

#### `openai`
- Keep `whisper-1` for timestamp-capable transcription.
- Keep `gpt-4o-transcribe` / `gpt-4o-mini-transcribe` available for general transcription, but do not assume they are the primary word-timing path.

#### `google`
- Keep as segment-level fallback.

#### `openrouter`
- Keep as fallback/experimental only.
- Do not position as the primary transcript timing backbone.

### Phase B: `mistral`

This is the cheapest meaningful addition because:
- `mistral` already exists in provider settings
- `mistral` already exists in validation
- `mistral` already exists in the shared provider system

Work required:
- add transcription model declarations
- add `MistralTranscriptionAdapter`
- expose to defaults/capability discovery

### Phase C: `deepgram`

This is the highest-value specialist provider to add.

Work required:
- add `deepgram` to `ProviderTypes`
- add settings card in `ProvidersTab`
- add validation in `ValidationService`
- add provider-specific transcription adapter
- expose in shared capability discovery
- route through defaults UI

Ship as ingest-first:
- do not build a general chat adapter unless there is a separate product need

### Phase D: `assemblyai`

Same vendor-level pattern as Deepgram:
- provider config
- settings UI
- validation
- transcription adapter
- capability discovery

Also ingest-first.
