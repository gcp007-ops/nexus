## Summary

Rebuild the lost ToolManager CLI-style prototype as a clean implementation on top of current `HEAD`, not by trying to recover the deleted worktree. The target product shape remains:

- `getTools`: discovery through a CLI-like selector string
- `useTools`: execution through one or more CLI-like command strings
- top-level context fields: `workspaceId`, `sessionId`, `memory`, `goal`, `constraints?`
- internal execution stays schema-driven and normalized to structured `{ agent, tool, params }`

The original source implementation appears to be gone. The surviving `codex/cli-tool-syntax` branch still contains the older JSON `context` / `request` / `calls` contract, so this work should be treated as a resurrection and re-port, not a merge.

The rebuild must also absorb newer changes that landed after the prototype work:

- chat media defaults for image/transcription tools
- built-in docs workspace prompt support
- live tool catalog injection into the prompt
- Perplexity subagent no-tools path
- current `getTools` discovery semantics and `subagent` exposure behavior

#### Questions to Resolve

- [ ] Should public multi-discovery and multi-execution use comma-separated `tool` strings, or should discovery support both comma-separated strings and an array alias?
- [ ] Should the parser auto-heal known top-level context flags accidentally placed inside `tool`, or should it fail hard with a targeted error?
- [ ] Should `getTools` return only CLI metadata, or return CLI metadata plus machine-readable schema behind a separate internal-only field?
- [ ] Should multiline content support be limited to quoted `\\n` sequences, or should flagged string params preserve raw substring content exactly?

#### Non-Goals

- Do not remove per-tool JSON schemas as the source of truth for validation.
- Do not expand this into a general shell language beyond the ToolManager boundary.

#### Design Approach

Use a strict boundary split:

1. **Public contract**
   The model sees and emits:
   - top-level context fields
   - a single `tool` string for discovery or execution

2. **Normalization layer**
   `ToolCliNormalizer` parses:
   - `getTools` selectors like `"storage"`, `"storage move"`, `"storage move, content read"`
   - `useTools` command strings like `"content read notes/today.md 1 40, storage move a.md archive/a.md"`

3. **Internal contract**
   Everything past normalization remains structured:
   - `getTools` still derives from real tool schemas
   - `useTools` still executes normalized `{ agent, tool, params }`
   - current validation and default-injection behavior remains intact

#### Public API Shape

`getTools`
```json
{
  "workspaceId": "default",
  "sessionId": "optional",
  "memory": "brief summary of the conversation so far",
  "goal": "brief statement of the current objective",
  "constraints": "optional rules or limits",
  "tool": "storage move, content read"
}
```

`useTools`
```json
{
  "workspaceId": "default",
  "sessionId": "optional",
  "memory": "brief summary of the conversation so far",
  "goal": "brief statement of the current objective",
  "constraints": "optional rules or limits",
  "tool": "content write --path note.md --content \"# Title\\n\\nBody\", storage open --path note.md --mode current --focus"
}
```

#### CLI Metadata Contract

`getTools` should return CLI-oriented metadata for each matched tool:

```typescript
interface CliToolSchema {
  agent: string;
  tool: string;
  description: string;
  command: string;
  usage: string;
  arguments: Array<{
    name: string;
    flag: string;
    type: string;
    required: boolean;
    positional: boolean;
    description?: string;
  }>;
  examples: string[];
}
```

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Public transport | JSON calls vs CLI-string facade | CLI-string facade | Smaller prompt surface and more bash-like ergonomics |
| Internal transport | CLI all the way down vs normalized structured IR | Structured IR | Keeps validation, tracing, and tool schema reuse |
| Multi-discovery | Repeated calls vs comma-separated selector string | Comma-separated selector string | Efficient and consistent with desired UX |
| Multi-execution | Repeated calls vs comma-separated command string | Comma-separated command string | Required by desired UX; still normalized internally |
| Error handling | Hard fail vs auto-heal for context flags | Start with hard fail + targeted error; optional heal later | Makes bugs obvious during rollout |
| Content parsing | Token-by-token only vs raw string preservation for flagged text | Preserve raw string for long text params | Needed for markdown/newlines and realistic note creation |

#### Compatibility Requirements

The resurrection must preserve these current behaviors:

- `DirectToolExecutor` media defaults for image/transcription flows
- `ToolBatchExecutionService.applyContextDefaults()`
- current `getTools` "empty tools means all tools for agent" behavior
- current `subagent` discovery visibility
- built-in docs workspace prompt section
- current Perplexity subagent no-tools branch

#### Implementation Sequence

1. Add `ToolCliNormalizer` with:
   - selector parsing
   - command parsing
   - schema-to-CLI metadata generation
   - stripping of common context params from per-tool schemas

2. Replace ToolManager public types:
   - `GetToolsParams`
   - `UseToolParams`
   - result types for CLI metadata

3. Port `getTools` to:
   - accept top-level context + selector string
   - support comma-separated multi-selector discovery
   - return CLI-only command metadata

4. Port `useTools` to:
   - accept top-level context + command string
   - support comma-separated multi-command execution
   - normalize into internal structured calls

5. Reconcile execution with current code:
   - preserve `DirectToolExecutor` media defaults
   - preserve `ToolBatchExecutionService.applyContextDefaults()`
   - preserve current progress observer wiring

6. Update prompting and instruction surfaces:
   - internal chat
   - subagents
   - external MCP connector

7. Update tool accordion/result shaping so multi-command `useTools` calls display inner actions clearly.

8. Add targeted debug logging for normalization failures during rollout, preferably to console rather than UI.
