# Implementation Plan: CLI Parser Edge-Case Followups

> Generated by Claude on 2026-04-19

> Branch: `fix/cli-parser-edge-cases`
> Builds on: PR #165 (`fix/cli-parser-family-audit`)

## Summary

Post-merge audit of PR #165 walked the parser systematically and found 5
real edge cases the reporter's 25-case characterization audit didn't cover.
All five are confirmed-reproducible silent failures or type-contract
violations on `fix/cli-parser-family-audit` HEAD. This PR ships them as a
single bundle since they all touch `ToolCliNormalizer.ts` and benefit from
landing together.

## Bugs

### High severity — silent corruption / contract violation

**EC-1: Non-bool flag silently consumes a following flag as its value**

```
content write --path --content "body"
→ params.path = "--content", positional "body" → content slot
```

`parseCommandSegment` line 574-580 grabs `tokens[index + 1]` as the value
without checking if that next token is itself a flag. Symmetric to the
C.2/C.3 bool-branch fix that landed in PR #165 — bool branch peeks before
consuming, non-bool branch trusts blindly.

**Fix**: in the non-bool flag branch, before consuming `next` as the value,
check `!next.wasQuoted && next.value.startsWith('--')`. If true, throw
`Flag "--X" requires a value, got flag "--Y"`. Quoted positional flag-like
strings (e.g., `--label "--important"`) still work because `wasQuoted=true`
is the explicit "this is data, not a flag" signal.

**EC-2: Unclosed quote silently captures rest of input**

```
content write "x.md" "unclosed
→ tokens = [..., {value: 'unclosed', wasQuoted: true}]   // silent
```

`tokenizeWithMeta` enters quote mode at line 183 and never re-emits if
the closing quote never arrives. End-of-input fallback at line 204 just
pushes whatever accumulated. In a multi-segment call, the unclosed quote
silently swallows subsequent commands.

**Fix**: at end-of-input, if `quote !== null`, throw
`Unclosed quote (started with X) in segment "..."`. Loud failure beats
silent corruption.

**EC-3: `array<number>` and `array<object>` always return strings**

```
ingest run --pages "1,2,3"     // schema declares array<number>
→ params.pages = ["1", "2", "3"]   // strings, downstream type error
```

`coerceValue` at line 226-238 only handles `array<string>` JSON-path
correctly (the `every(item => typeof item === 'string')` check). All
other element types fall through to `splitCsvRespectingQuotes` which
returns `string[]`. The contract says `array<X>` produces `X[]`.

**Fix**: extract item type from `array<X>`. After CSV/JSON split, run
`coerceValue(item, X)` per item. JSON-path: drop the
all-strings precondition and let the per-item coercer handle whatever
JSON.parse returned.

### Medium severity — surprising behavior

**EC-4: `Number("")` → 0 in number coercion**

```
content write --count ""
→ Number("") === 0
→ params.count = 0       // not undefined, not error
```

After D.2's empty-token emission landed in PR #165, empty quoted values
reach the coercer. JavaScript's `Number("")` is `0`, not `NaN`, so the
NaN guard doesn't catch it.

**Fix**: special-case `raw.trim() === ''` → return `raw` so schema
validation rejects it (or returns "missing required" for required slots).

**EC-5: `--flag=value` syntax not supported**

```
content write --path=x.md
→ Unknown flag "--path=x.md"
```

GNU long-option syntax (`--flag=value`). LLMs emit it. Current tokenizer
treats `=` as regular char, so the whole thing becomes one token, lookup
fails. Workaround is space-separated form (`--path x.md`).

**Fix**: in flag branch, before lookup, check if `normalizedFlag`
contains `=`. If yes, split into `(flagName, inlineValue)` and proceed
with the inline value. Booleans accept `--flag=true` / `--flag=false`
via the same code path as the C.2/C.3 peek.

## Out of scope (explicitly skipped)

- **Duplicate flag silent overwrite** — defensible "last wins" semantic,
  no consensus on behavior, low impact. File as separate issue if we
  want to debate.
- **`parseCliForDisplay` showing raw strings** — cosmetic only,
  accordion bubbles show string values pre-coercion. Not worth the
  schema-aware machinery to fix.
- **Other context fields beyond `workspaceId` (`sessionId`, `memory`,
  `goal`, `constraints`)** — open question I raised in the original
  plan; needs maintainer decision on whether tools should ever read
  these from `params` directly.
- **`splitCsvRespectingQuotes` mid-item quote handling** — underspecified
  but not broken. Add characterization tests later if we care.

## Risk

Low. Each fix is scoped to one branch in `parseCommandSegment` or
`coerceValue`. EC-3 is the only one that touches a code path tools
might rely on (array coercion); the all-strings precondition removal
in JSON-path is the only behavior change for existing callers, and
existing `array<string>` callers are unaffected because per-item
coercion of a string is a no-op.
