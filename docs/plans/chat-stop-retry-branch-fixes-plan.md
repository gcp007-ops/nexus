## Summary

Fix 12 bugs in the chat system affecting stop, retry, tool call display, and branch navigation. The bugs share 3 systemic root causes: (1) aggressive cleanup that unconditionally wipes tool calls, (2) shared mutable state with async timing issues during retry, and (3) a full re-render strategy that destroys live UI elements. The fix is organized into 4 phases: foundation fixes, incremental rendering, retry staging, and branch navigation cleanup.

#### Design Approach

**Incremental DOM Reconciliation** for `MessageDisplay`:
- Change `messageBubbles` from `MessageBubble[]` to `Map<string, MessageBubble>` keyed by message ID
- `setConversation()` reconciles: match by ID, reuse existing bubbles, create new, remove stale
- Keep `forceFullRender` flag for conversation switches (different conversation ID)
- Properly `unload()` removed bubbles

**Staging Pattern** for retry in `MessageAlternativeService`:
1. Show CSS loading overlay on existing message (do NOT clear content/tool calls)
2. Stream new response into local staging variable (not the conversation object)
3. On success: create branch with staged content, update `activeAlternativeIndex`, targeted UI update
4. On failure: remove overlay, original content untouched

**Tool Call Filtering** for `AbortHandler`:
- Preserve tool calls with `result !== undefined || success !== undefined`
- Only clear truly incomplete/pending tool calls
- Extract as reusable `filterCompletedToolCalls()` utility

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| State management pattern | A) Full immutable (Redux-style), B) Fix mutable with guards | B | Immutable rewrite is 5-10x effort; targeted guards address the bugs |
| MessageDisplay rendering | A) Full re-render, B) Incremental reconciliation | B | Full re-render is root cause of 4 bugs; existing `updateWithNewMessage()` supports incremental |
| Retry loading indicator | A) Clear content + show loading, B) CSS overlay on existing | B | Overlay prevents content flash and enables clean abort-restore |
| `messageBubbles` data structure | A) Array, B) Map<string, MessageBubble> | B | O(1) lookup vs O(n) linear scan; used 4+ times |
| Tool call filter location | A) Inline in AbortHandler, B) Shared utility | B | Same filter logic needed in both AbortHandler and retry |

#### Interface Contracts

**MessageDisplay reconciliation contract**:
- `setConversation(conversation)`: If same conversation ID, reconcile incrementally. If different ID, force full render.
- `updateMessage(messageId, message)`: Update a single bubble in-place (already partially exists).
- `cleanup()`: Calls `unload()` on ALL bubbles, clears the map.

**Retry operation contract**:
- Retry does NOT mutate live conversation object during operation
- Loading state shown via CSS overlay, not content clearing
- Branch creation + index update + UI refresh happens atomically after streaming completes

**Tool call persistence contract**:
- `convertAlternativesToEvent()` must persist: `id`, `type`, `function`, `name`, `parameters`, `result`, `success`, `error`, `executionTime`, `displayName`, `technicalName`
- Matches the main message serialization pattern (MessageRepository lines 185-196)
