# Implementation Plan: DRY Image Model Registry

> Generated by `/PACT:plan-mode` on 2026-02-26
> Status: PENDING APPROVAL

## Summary

Centralize image model definitions into a single `ImageModelRegistry` so that adding a new image model requires editing ONE file. Currently, model info is duplicated across 5 files (types, 2 adapters, tool schema, UI dropdown). The refactor creates a single registry with derived types, and all consumers (adapters, UI, tool schemas) read from it.

#### Design Approach

**Single cross-provider registry** (not per-provider files). Image models are inherently cross-provider — `gemini-2.5-flash-image` is available on both Google direct and OpenRouter. Unlike chat models where each provider has unique model families, image models share identity across providers. A single registry with provider mappings avoids duplicating entries.

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Registry structure | A) Extend existing ModelRegistry B) Per-provider files + aggregator C) Single ImageModelRegistry | C | Image models are cross-provider; per-provider would duplicate entries. Separate from ModelRegistry because image models have different shape (per-image pricing, reference images, etc.) |
| Type derivation | A) Manual union B) `as const` + `typeof` C) String type | B | Zero-duplication: `type ImageModel = typeof IMAGE_MODEL_SPECS[number]['id']` — adding a model to the registry auto-updates the type |
| Aspect ratio dimensions | A) In registry B) In adapters | B | Providers produce different pixel dimensions for the same ratio (Google: 2:3→768x1152, OpenRouter: 2:3→832x1248). Must remain provider-specific |

#### Interface Contracts

**`ImageModelSpec` type** (in `modelTypes.ts`):

```typescript
export interface ImageModelProviderMapping {
  provider: 'google' | 'openrouter';
  apiModelName: string;  // Provider-specific API name
}

export interface ImageModelSpec {
  id: string;                           // Internal ID: 'gemini-2.5-flash-image'
  name: string;                         // Display name: 'Nano Banana (Fast)'
  providers: ImageModelProviderMapping[];// Which providers offer this model
  maxReferenceImages: number;           // Max reference images (0 = no support)
  supportedImageSizes: string[];        // ['1K', '2K', '4K'] or ['512px', '1K', '2K', '4K']
  costPerImage: number;                 // USD per image
  contextWindow?: number;               // For text+image models (e.g., gpt-5-image: 400K)
  supportsReferenceImages: boolean;
  capabilities?: {
    supportsThinking?: boolean;
    supportsImageSearch?: boolean;
  };
}
```

**`ImageModelRegistry` static class**:

```typescript
export class ImageModelRegistry {
  static getAllSpecs(): ImageModelSpec[];
  static getSpec(modelId: string): ImageModelSpec | undefined;
  static getAllModelIds(): string[];
  static getModelsForProvider(provider: string): ImageModelSpec[];
  static getModelIdsForProvider(provider: string): string[];
  static getProviderApiName(modelId: string, provider: string): string | undefined;
  static getModelsForUI(provider: string): Array<{ id: string; name: string }>;
}
```

#### Implementation Sequence

**Step 1 — Foundation (no breaking changes)**
1. Add `ImageModelSpec` and `ImageModelProviderMapping` types to `modelTypes.ts`
2. Create `ImageModelRegistry.ts` with all 7 models:

| Model ID | Providers | Cost/Image |
|----------|-----------|------------|
| `gemini-2.5-flash-image` | google, openrouter | $0.039 |
| `gemini-3-pro-image-preview` | google, openrouter | $0.08 |
| `gemini-3.1-flash-image-preview` | google, openrouter | $0.04 |
| `gpt-5-image` | openrouter | $0.08 |
| `gpt-image-1` | (disabled) | — |
| `flux-2-pro` | openrouter | $0.05 |
| `flux-2-flex` | openrouter | $0.03 |

**Step 2 — Type derivation**
- Update `ImageTypes.ts`: derive `ImageModel` from registry via `as const` + `typeof`
- Removes the manually maintained type union

**Step 3 — Adapter refactoring (can be parallelized)**
- `GeminiImageAdapter.ts`: replace `supportedModels`, `maxReferenceImages`, pricing, `listModels()` with registry reads. Keep `aspectRatioToDimensions` (provider-specific pixel mappings) and `loadReferenceImages` (provider-specific API format)
- `OpenRouterImageAdapter.ts`: same treatment + replace `modelMap` with `ImageModelRegistry.getProviderApiName()`

**Step 4 — Consumer updates (can be parallelized)**
- `generateImage.ts`: dynamic schema enum from `ImageModelRegistry.getAllModelIds()`
- `ChatSettingsRenderer.ts`: replace hardcoded `IMAGE_MODELS` with `ImageModelRegistry.getModelsForUI(provider)`

Steps 2-4 are independent after Step 1 and can run in parallel.
