## Summary

Add a new Audio Editor leaf view to Claudesidian MCP that is transcript-first, workspace-scoped, and optimized for direct editing. The transcript is the primary surface for speech edits. A collapsible waveform/timeline dock sits below it: collapsed, it acts as a passive overview with play/pause; expanded, it becomes the full multitrack editing surface with pointer/cut mode, trims, fades, and gain keyframes.

This plan is grounded in:
- Research: `docs/research/audio-editor-leaf-view.md`
- Current UI prototype: `docs/mockups/audio-editor-leaf-view.html`
- Existing leaf pattern: `src/ui/tasks/TaskBoardView.ts`, `src/core/ui/TaskBoardUIManager.ts`, `src/ui/tasks/taskBoardNavigation.ts`
- Existing audio backend: `src/agents/apps/composer/services/AudioComposer.ts`, `src/agents/apps/composer/services/AudioMixer.ts`, `src/agents/apps/composer/services/AudioEncoder.ts`

The most important product decisions from the current iteration are:
- Workspace dropdown only. No project dropdown in the v1 leaf UI.
- Left sidebar / right canvas layout.
- Sidebar shows a searchable folder tree of sound files from the workspace root.
- Transcript is editable directly; do not over-index on button-heavy editing.
- Collapsed dock is just `play/pause + overview waveform`.
- Expanded dock exposes the full timeline and editing controls.
- Transcript overlays show attached sound file names only; do not visually classify voice/music/SFX in the UI.
- Timeline clips use the audio file name by default. Renamed clip labels are a separate later layer.

- The repo’s durable storage pattern is already established: append-only JSONL events via `JSONLWriter`, SQLite cache via repositories + sync appliers, and service facades on top. Audio editor persistence should fit that system rather than writing bespoke files.
- The current transcription implementation is too tightly coupled to ingestion. For the audio editor, transcription should become its own domain service and type system, with ingest calling into it rather than owning provider/model orchestration.

#### Design Approach

**Leaf runtime pattern**
- Mirror the Task Board structure exactly where it fits:
  - dedicated `AUDIO_EDITOR_VIEW_TYPE`
  - `audioEditorNavigation.ts` with open modes and normalized state
  - `AudioEditorUIManager.registerViewEarly()`
  - `AudioEditorView extends ItemView`
  - async `initializeView()` with service retry instead of assuming startup order
  - `getState()` / `setState()` for leaf persistence

**State split**
- `ItemView` state should store ephemeral UI only:
  - selected workspace/project
  - selected clip/track
  - expanded/collapsed dock
  - pointer vs cut mode
  - zoom and local viewport
- Durable editor state should live in workspace-scoped storage:
  - tracks
  - clips
  - transcript blocks/anchors
  - fades
  - keyframes / automation points
  - source file references

**Save semantics**
- Direct manipulation should update in-memory state continuously, but persist only on commit boundaries:
  - transcript text: debounce on pause/blur
  - drag trim/fade/keyframe/move: persist once on pointer-up
  - keyboard actions (`delete`, `cut`, `new track`, `insert source`): persist immediately as one logical operation
- This avoids flooding the append-only log with per-pixel drag events while still giving deterministic edit history.

**Playback/render split**
- Live editor playback should use a real-time `AudioContext`.
- Export should use `OfflineAudioContext`.
- Both should be driven by the same graph-building/compiler layer so preview and export obey the same trim/fade/automation rules.

**Transcription architecture**
- Transcription should be pulled out of ingest and treated like image generation: a dedicated service domain with its own provider/model types, provider capability discovery, and orchestration layer.
- Ingest should become one consumer of that domain, not the place where transcription provider logic lives.
- Audio editor, ingest, and any future transcript tools should all call the same transcription service surface.

**Primary editing model**
- Transcript drives speech editing.
- Timeline remains the source of truth for multitrack structure.
- Non-speech media is attached to transcript ranges as lightweight overlays, but those overlays remain timeline clips underneath.

**Leaf layout**
```
Sidebar
├── Workspace dropdown
├── Search
└── Folder tree of sound files

Canvas
├── Transcript editor (primary)
└── Timeline dock
    ├── Collapsed: play/pause + overview
    └── Expanded: mode toggle + full multitrack editing
```

**Dock states**
- **Collapsed**
  - Play/pause
  - Passive overview waveform
  - Follows transcript focus and playback position
- **Expanded**
  - Pointer mode
  - Cut mode
  - Add keyframe
  - New track
  - Zoom
  - Multitrack clip editing

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Workspace selector | Workspace + project vs workspace only | Workspace only | Less top-level chrome; aligns with workspace-scoped file browser |
| Source browser | Flat list vs folder tree | Folder tree | Preserves vault organization and makes workspace scoping legible |
| Primary editing surface | Timeline-first vs transcript-first | Transcript-first | Matches speech/audio editing workflow goal |
| Timeline collapsed state | Hide completely vs mini overview | Mini overview | Maintains orientation during transcript editing |
| Timeline controls | Always visible vs expanded-only | Expanded-only except play/pause | Reduces visual noise in default state |
| Media overlays in transcript | Categorized badges vs plain file names | Plain file names | Avoid fake precision; all are just sound files in v1 |
| Clip labels | File name vs renamed clip title only | File name by default, optional alias later | Simple mental model, matches transcript overlay labels |
| Editing interaction | Button-heavy vs direct manipulation | Direct manipulation | Lower click count; closer to editor behavior |
| Track metadata | Fixed track labels vs inline editable names | Inline editable names | User must be able to name tracks freely |

#### Interface Contracts

```typescript
interface AudioEditorViewState extends Record<string, unknown> {
  workspaceId?: string;
  projectId?: string; // internal persistence only, not top-level UI
  selectedTrackId?: string;
  selectedClipId?: string;
  selectedTranscriptRange?: {
    paragraphId: string;
    startOffset: number;
    endOffset: number;
  };
  timelineExpanded?: boolean;
  timelineMode?: 'pointer' | 'cut';
  sourceSearch?: string;
}

interface AudioClip {
  id: string;
  sourceFile: string;
  clipLabel?: string;        // optional alias; defaults to source file name
  timelineStart: number;
  sourceStart: number;
  sourceEnd: number;
  fadeIn: number;
  fadeOut: number;
  keyframes: Array<{ id: string; t: number; value: number }>;
  transcriptAnchorIds?: string[];
}

interface TranscriptOverlay {
  sourceFile: string;        // display this by default
  clipId: string;
}
```

**Naming contract**
- Transcript overlays display `basename(sourceFile)`.
- Timeline clips display `clipLabel ?? basename(sourceFile)`.
- If clip rename is added later, it must not overwrite the original source file name.

**Transcript contract**
- V1 durable transcript shape should be paragraph/block based, with optional word anchors:

```typescript
interface TranscriptBlock {
  id: string;
  start: number;
  end: number;
  text: string;
  speaker?: string;
  sourceClipIds: string[];
  wordIds?: string[];
}

interface TranscriptWord {
  id: string;
  blockId: string;
  text: string;
  start: number;
  end: number;
  clipId: string;
}
```

- Segment-only transcription is enough for paragraph navigation.
- Word-directed editing (`highlight a word and delete it`) requires either:
  - extending the speech-API transcription path to request word timings where supported, or
  - an explicit post-transcription alignment pass.
- Google/OpenRouter multimodal transcription currently returns plain text only in this repo, so it does not satisfy word-editing requirements by itself.

**Automation contract**
- Keyframes should be clip-local, not track-global, because trims/splits must move their automation with the clip.
- Store automation in clip-local seconds from clip start:

```typescript
interface ClipGainPoint {
  id: string;
  clipId: string;
  offsetSeconds: number;
  gain: number; // linear gain, e.g. 0.0-2.0
}
```

- On split:
  - left clip keeps points before split
  - right clip gets points at/after split with offset rebased to the new clip start
- On left-trim:
  - points before the new start are dropped
  - surviving points are shifted left by the trim delta

**Render compiler contract**
- The current `AudioMixer` input shape is too small for the mockup’s editing model.
- Add an editor-side compiler layer that maps `AudioProject` → renderable clips → audio graph.
- Each clip should get its own `AudioBufferSourceNode` and `GainNode`.
- Fades and automation should be scheduled with `setValueAtTime()` / `linearRampToValueAtTime()`.
- Export can still terminate in the existing `AudioEncoder`.

## Current UX Contract

These decisions are intentionally locked in unless a later plan supersedes them:

- The top-level UI should **not** have both workspace and project selectors.
- The leaf should read as **left sidebar + right canvas**.
- Sidebar should show sound files **as folders**, not a flat list.
- Search filters both folder paths and filenames.
- Transcript is the default working surface.
- Collapsed dock is **not** an editing surface; it is orientation + transport.
- Expanded dock is the editing surface.
- Overlapping non-speech clips are represented in transcript as lightweight overlay rows/chips.
- Overlay text should use the **audio file name**.
- Timeline segment text should use the **audio file name by default**.
- Track names remain user-editable.
- Avoid visually classifying assets as voice/music/SFX in the v1 UI.

  - [TranscriptionService.ts](/Users/jrosenbaum/Documents/Code/.obsidian/plugins/claudesidian-mcp/src/agents/ingestManager/tools/services/TranscriptionService.ts)
- Practical consequence:
  - adding a new **model** to an existing provider is relatively cheap
  - adding a new **vendor** requires provider settings, validation, capability surfacing, and transcription-service integration

### Recommended Refactor
- Pull transcription out of ingest and model it after the image-generation architecture:
  - shared `TranscriptionService`
  - `VoiceTypes` for provider/model/result shapes
  - provider-specific transcription adapters
  - shared capability discovery for “default transcription”
- Then make ingest a caller of that service, not the owner of transcription provider branching.
- This gives one consistent surface for:
  - vault ingest
  - audio editor transcript generation
  - future transcript repair / alignment tools

### Why This Is Better
- It matches the repo’s direction for image generation: dedicated orchestration, dedicated types, dynamic provider/model discovery.
- It avoids duplicating provider logic between ingest and audio editor.
- It lets “Default transcription” remain one global setting, backed by a real shared service instead of an ingest-only catalog.
- It creates a clean place for:
  - word timestamps
  - diarization
  - transcript prompts / glossaries
  - alignment-related metadata

### Provider Capability Snapshot
- `openai`
  - already integrated
  - best fit here is `whisper-1` when word timestamps are required
  - `gpt-4o-transcribe` is still useful for general STT, but not the best default for word-level editing in this repo
- `groq`
  - already integrated
  - strongest current default for word-level timestamp work in the existing system
  - should remain the default editor transcription provider for v1
- `google`
  - already integrated
  - good segment-level transcript fallback
  - not sufficient by itself for word-directed editing in the current implementation
- `openrouter`
  - keep as experimental/fallback only
  - useful for multimodal transcription experiments, but not a stable core timing backbone
- `mistral`
  - high-priority addition
  - already exists as a configured provider in the shared provider system, so this is primarily an ingest/transcription extension
  - good candidate for segment + word timestamp + diarization path
- `deepgram`
  - high-value specialist STT provider
  - best added as a new provider phase after Mistral
  - strongest upside: word timing, utterances, diarization, keyword biasing
- `assemblyai`
  - also strong, but second after Deepgram
  - strong fit for word timestamps, speaker labels, and transcript cleanup features

### Phased Recommendation

**Phase A: Harden existing providers**
- Keep `groq` as the default word-timestamp path.
- Keep `openai` with `whisper-1` as the second timestamp-capable path.
- Keep `google` and `openrouter` as segment-level fallbacks.
- Extend the current transcription model to preserve `words[]` when available.

**Phase B: Add Mistral to ingest transcription**
- Since `mistral` already exists in the shared provider system, add it by:
  - extending `IngestModelCatalog`
  - extending `TranscriptionService`
  - exposing it in `IngestCapabilityService`
- This phase does **not** require new provider settings UI or a new adapter-registry concept.

**Phase C: Add Deepgram as a new provider**
- Add `deepgram` provider config in `ProviderTypes`.
- Add settings card + validation in `ProvidersTab` / `ValidationService`.
- Add ingest transcription model declarations and execution branch.
- Treat it as an ingest-first provider; only add a chat adapter later if we have a real use case.

**Phase D: Add AssemblyAI as a new provider**
- Same vendor-level work as Deepgram.
- Also ingest-first unless a broader provider use case appears.

### Recommended Ranking For The Audio Editor
1. `groq`
2. `openai` (`whisper-1` path for words)
3. `mistral`
4. `deepgram`
5. `assemblyai`
6. `google`
7. `openrouter`

## Persistence Model

### Recommended Entity Shape
- `audio_projects`
  - one row per saved timeline/document
  - workspace-scoped
  - stores name, sample rate, duration, metadata JSON
- `audio_tracks`
  - project-scoped
  - stores name, order, and future track flags
- `audio_clips`
  - track-scoped
  - stores source file path, timeline start, source start/end, fades, optional alias label, transcript anchor IDs
- `audio_automation_points`
  - clip-scoped
  - stores offset seconds + gain value
- `audio_transcript_blocks`
  - project-scoped
  - stores text and timing ranges
- `audio_transcript_words`
  - optional in schema rollout, but required for true word-directed edits

### Event Log
- Follow the task-management pattern:
  - `audio_project_created`
  - `audio_project_updated`
  - `audio_track_added`
  - `audio_track_updated`
  - `audio_track_deleted`
  - `audio_clip_added`
  - `audio_clip_updated`
  - `audio_clip_deleted`
  - `audio_clip_split`
  - `audio_automation_point_added`
  - `audio_automation_point_updated`
  - `audio_automation_point_deleted`
  - `audio_transcript_imported`
  - `audio_transcript_updated`

### Save Boundaries
- Persist on semantic edits, not every motion sample.
- During drag:
  - keep local mutable draft state
  - commit one repository update on release
- During text editing:
  - maintain local buffer
  - commit after debounce or blur
- This keeps JSONL event history human-reasonable and sync-safe.

## Keyframes

### Why Current Mixer Is Not Enough
- [AudioMixer.ts](/Users/jrosenbaum/Documents/Code/.obsidian/plugins/claudesidian-mcp/src/agents/apps/composer/services/AudioMixer.ts) only knows:
  - file
  - track offset
  - track volume
  - fade in
  - fade out
- It does not know:
  - clip trims
  - multiple clips on one track
  - per-clip automation points
  - clip splitting behavior

### Recommended Implementation
- Introduce `AudioRenderCompiler` that walks the editor project model and emits renderable clips.
- For each clip:
  - decode source audio
  - trim to `sourceStart/sourceEnd`
  - place it at `timelineStart`
  - attach a dedicated `GainNode`
  - schedule fade ramps
  - schedule automation keyframes
- Use the same compiler for:
  - live preview (`AudioContext`)
  - export (`OfflineAudioContext`)

### Editing Rules
- Keyframes belong to clips, not tracks.
- Dragging a clip moves its keyframes with it.
- Splitting a clip repartitions its keyframes.
- Deleting transcript words that map to a clip range should also update or drop overlapping automation points inside the removed region.
