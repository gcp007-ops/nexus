# Implementation Plan: LLM Tool Evaluation Harness

> Generated by `/PACT:plan-mode` on 2026-04-15
> Status: IN_PROGRESS

## Summary

Build a generalizable LLM evaluation harness that exercises the full production streaming + tool continuation code path against real LLM APIs. The harness uses declarative TypeScript scenarios to define multi-turn tool call interactions, injects a mock `IToolExecutor` at the existing injection point in `StreamingOrchestrator`, and captures exact request/response bodies for debugging provider-specific issues. Zero production code changes required.

### Architecture Phase
**Effort**: Low

#### Components

| Component | Type | Purpose |
|-----------|------|---------|
| `EvalToolExecutor` | New class | `IToolExecutor` implementation with configurable responses + call capture |
| `EvalAdapterRegistry` | New class | Minimal `IAdapterRegistry` wrapping pre-constructed adapters |
| `EvalRunner` | New class | Orchestrates scenario execution via real `StreamingOrchestrator` |
| `RequestCapture` | New utility | Wraps `__setRequestUrlMock` to capture request/response bodies |
| `EvalScenario` / `EvalStep` | New types | Declarative scenario definition format |
| Assertion helpers | New utilities | Tool call matchers, text content matchers |

#### Design Approach

**Strategy + Pipeline pattern** mirroring the production code's own patterns:

```
EvalScenario → EvalRunner → StreamingOrchestrator (real) + EvalToolExecutor (mock)
                                    ↓
                         ToolContinuationService (real)
                                    ↓
                         ProviderMessageBuilder (real)
                                    ↓
                         BaseAdapter.generateStreamAsync (real → real API)
```

The key insight: `StreamingOrchestrator` already accepts `IToolExecutor?` as a constructor param. We inject `EvalToolExecutor` — everything else runs on real production code with real API calls. The tool execution is the only mocked layer.

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Test runner | Jest / Custom / Vitest | Jest | Already configured, existing patterns, describe.each for model matrix |
| Scenario format | TypeScript / YAML / JSON | TypeScript | Type checking, IDE support, no parser dependency |
| Mock injection point | IToolExecutor / HTTP level / Adapter wrapper | IToolExecutor | Clean interface, already the decoupling point, tests real continuation code |
| Adapter setup | Real AdapterRegistry / Minimal bypass | Minimal `EvalAdapterRegistry` | Avoids async init, OAuth, dynamic imports — just wraps pre-built adapters |
| Test level | Adapter-only / Full orchestrator | Full orchestrator | Tests complete production path including pingpong, tool merging, iteration limits |

#### Interface Contracts

**Two execution modes** — configurable in `eval.config.ts`:

| Mode | Tool Execution | Speed | Use Case |
|------|---------------|-------|----------|
| `mock` | `EvalToolExecutor` returns scripted responses | Fast (~5s/scenario) | Development, CI, regression testing |
| `live` | `LiveToolExecutor` runs real Nexus agents against a test vault | Slower (~15-30s/scenario) | Full E2E validation, system prompt tuning, real response verification |

In **live mode**, the harness initializes the real agent stack (ToolManager, ContentManager, StorageManager, etc.) against a test vault directory (`tests/eval/test-vault/`). Tool calls go through real agents operating on real files. Responses are actual Nexus output, not mocks. The test vault is reset between scenarios from a fixture snapshot.

This means you can test things like: "Ask the LLM to read meeting.md, summarize it, and write summary.md" — and verify that `summary.md` actually exists in the test vault with the right content afterward.

**EvalConfig** — top-level config (loaded from `tests/eval/eval.config.ts`, no code changes needed to adjust):
```typescript
interface EvalConfig {
  /** Provider configurations — easy to swap models/providers */
  providers: {
    [key: string]: {           // e.g. 'openrouter', 'anthropic', 'openai-codex'
      apiKeyEnv: string;       // env var name, e.g. 'OPENROUTER_API_KEY'
      models: string[];        // e.g. ['anthropic/claude-sonnet-4.6', 'openai/gpt-5.4-mini']
      enabled: boolean;        // toggle providers without removing config
    };
  };
  /** Execution mode */
  mode: 'mock' | 'live';      // mock = scripted responses, live = real agents + test vault
  /** Test vault path (live mode only) */
  testVaultPath?: string;      // default 'tests/eval/test-vault/'
  /** Global defaults (overridable per-scenario) */
  defaults: {
    temperature: number;       // default 0, adjustable
    maxRetries: number;        // default 1, adjustable
    retryDelayMs: number;      // default 2000
    timeout: number;           // per-scenario timeout in ms, default 120_000
    systemPrompt: string;      // use actual Nexus system prompt by default
  };
  /** Request/response capture settings */
  capture: {
    enabled: boolean;          // toggle capture on/off
    dumpOnFailure: boolean;    // auto-dump to test-artifacts/ on failure
    artifactsDir: string;      // default 'test-artifacts/'
  };
  /** Which scenario files to run (glob pattern) */
  scenarioGlob: string;        // default 'tests/eval/scenarios/**/*.eval.ts'
}
```

**EvalScenario** — declarative scenario definition:
```typescript
interface EvalScenario {
  name: string;
  description: string;
  /** Override which providers/models to test (defaults to all from config) */
  providers?: string[];
  models?: string[];
  /** Override config defaults per-scenario */
  overrides?: {
    temperature?: number;
    maxRetries?: number;
    timeout?: number;
    systemPrompt?: string;    // override the default system prompt
  };
  tools: Tool[];               // Production Tool type
  turns: EvalTurn[];           // Multi-turn conversation flow
}

interface EvalTurn {
  userMessage?: string;        // Only on first turn
  expectedTools: ExpectedToolCall[];
  mockResponses: Record<string, MockToolResponse>;
}

interface ExpectedToolCall {
  name: string;                // Exact tool name match
  params?: Record<string, unknown>;  // Partial match via objectContaining
  optional?: boolean;          // Model may or may not call this
}

interface MockToolResponse {
  success: boolean;
  result?: unknown;
  error?: string;
}
```

### Test Phase
**Effort**: Low (the harness IS the test infrastructure)

#### Assertion Strategy (4 layers)

| Layer | Type | Example |
|-------|------|---------|
| Structural | MUST pass | Tool call count in expected range, tool names in expected set |
| Semantic | SHOULD pass | Param values match via `objectContaining` |
| Behavioral | Verify outcome | Final response contains expected keywords |
| Anti-assertions | MUST NOT | No hallucinated tool names, no empty tool call arrays |

#### Test Scenarios (Priority)

**P0 — Core Flows (must work for the plugin to function)**
| Scenario | Type | What it tests |
|----------|------|---------------|
| Single tool call round-trip | Integration | Basic: prompt → tool call → mock result → text response |
| Multi-turn sequential (3+ rounds) | Integration | Read → write → move note chain |
| getTools → useTools two-tool flow | Integration | The actual Nexus architecture end-to-end |
| Real system prompt with workspace context | Integration | Production system prompt produces correct tool behavior |

**P1 — Reliability (users will hit these)**
| Scenario | Type | What it tests |
|----------|------|---------------|
| Ambiguous prompts ("do something with my notes") | Integration | Model picks reasonable tool despite vagueness |
| Tool returns error → model recovers | Integration | Error handling in continuation flow |
| Tool call ID consistency across rounds | Integration | IDs match between assistant tool_calls and tool results |
| Provider parity (same scenario, N models) | Integration | Behavioral consistency across providers |
| Large tool response payload | Integration | Big JSON results don't break continuation |
| Model calls wrong/nonexistent tool | Integration | Graceful handling when model hallucinates a tool name |

**P2 — Edge Cases & Hardening**
| Scenario | Type | What it tests |
|----------|------|---------------|
| Tool iteration limit reached (15+) | Integration | Safety cap works, user notified |
| Model refuses to call tools (responds with text) | Integration | Graceful fallback when model ignores tools |
| Empty/null tool arguments | Integration | Continuation doesn't break on malformed args |
| System prompt variations | Integration | Different prompts produce different tool routing |
| Concurrent tool calls in one turn | Integration | Model returns 2+ tool_calls at once |

#### Non-Determinism Mitigations
- `temperature` configurable per-scenario (defaults from config)
- System prompt sourced from actual Nexus production prompt
- Flexible assertions (match tool name + required params, not exact arg strings)
- Retries configurable per-scenario (defaults from config)
