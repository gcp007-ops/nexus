## Summary

Local `main` has 452 files modified as part of a lint + type-safety pass. The goal was correct — replacing `any` with `unknown` is the right direction — but the migration was incomplete. The result: **578 TypeScript compile errors** across 120 files and **72 test failures** across 7 suites. The plugin does not build.

This plan completes the migration properly: adding the type guards, narrowing casts, and interface fixes that `unknown` requires at every callsite.

**Audit findings**: `npm run build` → 578 errors. `npm run test` → 72 failures, 1479 passing.

---

## Root Cause Analysis

Three root causes cascade into most of the 578 errors:

### RC-1: `ValidationSchema [key: string]: unknown` cascade
`src/utils/validationUtils.ts` added a `[key: string]: unknown` index signature to `ValidationSchema`. This means every property access on a `ValidationSchema` (e.g., `schema.properties`, `schema.required`) now returns `unknown` instead of the declared type. The internal functions in `validationUtils.ts` itself were also changed to take `unknown` params without adding narrowing — causing 39 errors in the file and cascading into 4 schema builder files via a broken import path.

### RC-2: `BaseAdapter` — `StreamParseOptions` vs `SSEStreamOptions` mismatch
`src/services/llm/adapters/BaseAdapter.ts` defined a new local `StreamParseOptions` interface whose callbacks use `JsonObject = Record<string, unknown>`. But `SSEStreamProcessor.processSSEStream` expects `SSEStreamOptions` whose callbacks use `SSEParsedEvent`. The two are structurally incompatible, causing 30 errors in `BaseAdapter` that cascade into all 8+ adapter files.

### RC-3: Incorrect import paths in schema builders
Four files in `src/utils/schemas/builders/` import `'../../../validationUtils'` — which resolves to `src/validationUtils.ts` (does not exist). The correct path is `'../../validationUtils'` (`src/utils/validationUtils.ts`).

### Phase 1 — Root Causes

#### Task 1.1: Fix `validationUtils.ts` internal narrowing
**File**: `src/utils/validationUtils.ts`
**Errors**: 39

The `validateParams(params: unknown, schema: unknown)` function narrows `schema` to `object` via `typeof schema !== 'object'` check, then accesses `schema.required`, `schema.properties`, etc. — which don't exist on `object`. Fix:

```typescript
// After the typeof guard, narrow to ValidationSchema
if (!schema || typeof schema !== 'object') return errors;
const typedSchema = schema as ValidationSchema;

// Use typedSchema throughout
if (typedSchema.required && Array.isArray(typedSchema.required)) { ... }
if (typedSchema.properties) {
  validateObjectProperties(params, typedSchema.properties, [], errors);
}
```

Apply the same pattern to all internal helpers:
- `validateObjectProperties(obj: unknown, ...)` — add `const typedObj = obj as Record<string, unknown>;` after a `typeof obj === 'object' && obj !== null` guard
- `validateType(value: unknown, ...)` — use `typeof value` checks before accessing value properties
- `validateAllOf`, `validateAnyOf`, `validateOneOf` — accept `ValidationSchema[]` not `unknown[]` (the cast at the entry point is enough)
- `generateParameterHints`, `enhanceSchemaDocumentation` — cast `schema` to `ValidationSchema` at entry, then use `Object.entries(typedSchema.properties ?? {})`

For `Object.entries<any>(schema.properties)` calls — replace with `Object.entries(typedSchema.properties ?? {}) as [string, ValidationSchema][]`.

// Before: indexing returns unknown

Use casts (`as T`) for internally-typed data where you control the schema. Use guards for externally-sourced data (API responses, JSON parsing).

#### Task 5.3: Fix `loadDesktopModule` for Jest environment
**Failures fixed**: ~20 (cliProcessRunner, ClaudeHeadlessService, GoogleGeminiCliAdapter, AnthropicClaudeCodeAdapter)

`src/utils/cliProcessRunner.ts` changed to use `loadDesktopModule('child_process')` which checks `globalThis.require`. In Jest, `require` is module-scoped, not on `globalThis`, so it throws "Desktop module loader is unavailable."

Fix in `src/utils/cliProcessRunner.ts`:
```typescript
function loadDesktopModule<TModuleName extends keyof CliProcessRunnerDesktopModuleMap>(
  moduleName: TModuleName
): CliProcessRunnerDesktopModuleMap[TModuleName] {
  if (!Platform.isDesktop) {
    throw new Error(`${moduleName} is only available on desktop.`);
  }

  // Fall back to module-scoped require for Node/Jest environments
  const maybeRequire = (globalThis as typeof globalThis & {
    require?: (moduleId: string) => unknown;
  }).require ?? require;  // ← add `?? require` fallback

  if (typeof maybeRequire !== 'function') {
    throw new Error('Desktop module loader is unavailable.');
  }

  return maybeRequire(moduleName) as CliProcessRunnerDesktopModuleMap[TModuleName];
}
```

## Guiding Principles for This Migration

**Use `as CastType` for internal data** (data you own and control):
```typescript
const value = record[key] as string; // You know it's always a string here
```

**Use type guards for external data** (API responses, user input, JSON.parse):
```typescript
function isString(v: unknown): v is string { return typeof v === 'string'; }
if (isString(response.content)) { use(response.content); }
```

**Narrow at the boundary, not at every callsite**. If a function takes `unknown`, narrow it once at the top and pass typed values to helpers — don't add `as T` casts at 20 individual property accesses.

**`unknown` is better than `any`**. This migration is correct in intent. The goal is to complete it, not revert it.
