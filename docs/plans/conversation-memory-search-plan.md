# Implementation Plan: Conversation Memory Search

> Generated by `/PACT:plan-mode` on 2026-02-07
> Status: IN_PROGRESS

## Summary

Enhance `searchMemory` with semantic search across conversation turns and tool call traces. All QA pairs (user+assistant turns and tool invocation+result pairs) are embedded into sqlite-vec for vector similarity search. The tool operates in two modes: **Discovery** (broad search within workspace, returns snippets + sessionId) and **Scoped** (search within workspace+session, returns N-turn window around matches). Backfill embeds all existing conversations on first enable.

**Additionally**, introduce a **chunking strategy** for conversation embeddings. Q and A are chunked independently (1800 chars/chunk with 200 char overlap), all chunks share a `pairId`, and any chunk match returns the **full QA pair** to the LLM. Chunks are indexing artifacts only — retrieval always returns complete content. Note embeddings are left as-is (current 2000-char truncation aligns with the model's 512-token limit).

- **Tool call format**: Template to natural language: `Tool: {name}({key_param}: {value}) -> {success|error}: {result_summary}`

#### Design Approach

**Content Chunking — Two Strategies**:

**For QA Pairs** (two linked parts: question + answer):
Q and A are chunked **independently**, but all chunks share a `pairId`. Chunks are for search indexing only — on any chunk match, the **full QA pair** (complete Q + complete A) is returned. The embedding is the "finder", the full content is the "reader".

```
Example: 10K-word user question + 20K-word assistant response
  Q chunks: chunk(user_content) → chunks Q0, Q1, Q2...  (all share pairId="abc")
  A chunks: chunk(assistant_content) → chunks A0, A1, A2...  (all share pairId="abc")

Search hit on chunk A5 → look up pairId "abc" → return full Q + full A
```

Short QA pairs (both Q and A ≤500 chars) produce just 2 chunks total (one Q, one A) — minimal overhead.

**Unified QA Pair Model**: Both conversation turns and trace pairs are embedded identically — Q and A chunked independently, all chunks linked by `pairId`, stored in `conversation_embeddings` vec0 table with metadata distinguishing `pairType: 'conversation_turn' | 'trace_pair'` and `side: 'question' | 'answer'`.

**Two-Mode Search**:
- **Discovery** (no sessionId): KNN vector search across workspace → returns top-K with full QA pairs + sessionId + conversationId + sequence location
- **Scoped** (with sessionId): KNN vector search filtered by sessionId → for each match, return full QA pair + `ConversationWindowRetriever` fetches N turns around the match

**Data Flow — Real-time**:
```
Message state → 'complete' (role='assistant')
  → ConversationEmbeddingWatcher observes event
  → Look up previous user message by sequenceNumber
  → Generate pairId (e.g., conversationId + startSeqNum)
  → Chunk Q independently: ContentChunker.chunk(user_content)
  → Chunk A independently: ContentChunker.chunk(assistant_content)
  → For each Q chunk: embed with metadata(pairId, side='question', chunkIndex)
  → For each A chunk: embed with metadata(pairId, side='answer', chunkIndex)
  → Store full Q and A content for retrieval (in messages table, already exists)
```

**Data Flow — Search**:
```
searchMemory(query, workspaceId, sessionId?)
  → MemorySearchProcessor.searchConversationEmbeddings()
  → EmbeddingService.semanticConversationSearch(query, workspaceId, sessionId?, limit)
  → KNN search (fetch 3x limit for reranking headroom)
  → Deduplicate by pairId (keep best-matching chunk per pair)
  → Rerank with multi-signal scoring (see below)
  → For each unique pairId: fetch full Q + full A from messages table
  → In Scoped mode: also fetch N-turn window via ConversationWindowRetriever
  → Returns ConversationSearchResult[] (with full QA pair content, reranked)
```

**Reranking Strategy** (extends existing EmbeddingService reranking pattern):

Fetch 3x candidates from KNN, then apply multi-signal scoring before slicing to `limit`:

| Signal | Boost | Decay | Applied As |
|--------|-------|-------|------------|
| **Semantic distance** | Base score | — | `vec_distance_l2` (lower = better) |
| **Recency** | 20% max | 14-day linear decay | `score * (1 - recencyBoost)` |
| **Session density** | 15% max | — | `score * (1 - densityBoost)` when 2+ hits in same session |
| **Note references** | 10% | — | `score * 0.9` when QA pair contains `[[note]]` matching query terms |

**Signal Details**:
- **Recency**: Linear decay matching trace pattern — conversations < 14 days old get up to 20% boost. Formula: `0.20 * (1 - daysSince/14)`.
- **Session density**: When multiple QA pairs from the same session appear in the candidate set, boost all of them. Formula: `0.15 * min(1, (sessionHitCount - 1) / 3)`. This surfaces conversations where a topic was discussed extensively.
- **Note references**: If Q or A content contains `[[note_name]]` wiki-links, and any of the query terms appear in those note names, apply a 10% boost. This creates a lightweight graph connection between conversations and notes without requiring a full graph index.

All boosts applied multiplicatively to L2 distance (lower = better), matching existing `EmbeddingService.semanticSearch()` pattern at lines 238-268.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Content handling for long text | A: Truncate (current), B: Chunk with overlap | B: Chunk | No information loss; all content searchable; fixes existing note embedding gap |
| QA pair chunking strategy | A: Concatenate Q+A then chunk, B: Chunk Q and A independently with shared pairId | B: Independent | Preserves semantic coherence per side; any chunk hit returns full pair; no lost context |
| Chunk size | A: 500 chars, B: 1000 chars, C: 1800 chars | A: 500 chars | Small chunks = better search precision. Full pair returned on match regardless of chunk size. ~125 tokens per chunk — well within model capacity. |
| Chunk overlap | A: 0 (none), B: 100 chars, C: 200 chars | B: 100 chars | Prevents splitting semantic units at boundaries; minimal storage overhead |
| Search result content | A: Return matching chunk only, B: Return full QA pair | B: Full pair | Chunks are indexing artifacts; the LLM needs complete Q+A context to understand the match |
| Extend EmbeddingService vs. separate pipeline | A: Extend existing, B: New service | A: Extend | Reuses singleton engine, identical vec0 pattern, avoids duplicating init logic |
| Separate vec0 table vs. shared | A: Separate `conversation_embeddings`, B: Share `trace_embeddings` | A: Separate | Different metadata shape, lifecycle, search patterns; performance isolation |
| QA pair reference model | A: Separate turn ID, B: Sequence number range | B: Range (`startSeq + endSeq`) | Sequence numbers are canonical ordering; handles multi-message tool chains |
| Windowed retrieval approach | A: Single JOIN query, B: Two-step (KNN then window fetch) | B: Two-step | Preserves vec0's optimized KNN path; window fetch uses existing index |
| Backfill strategy | A: One-time migration, B: Background job | B: Background via IndexingQueue | Non-blocking; resume-on-interrupt; existing pattern proven |
| Conversations without workspace | A: Skip, B: Embed with NULL workspaceId | B: Embed with NULL | Include in unscoped discovery; filter in scoped mode |
| Branch conversations | A: Embed, B: Skip initially | B: Skip | Main conversations first; branches added later |
| Search result deduplication | A: Return all chunks, B: Dedupe by pairId | B: Dedupe by pairId | Same pair's Q and A chunks may both match; dedupe to single result, return full pair |

#### Interface Contracts

**Content Chunker**:
```typescript
interface ChunkOptions {
  maxChunkSize: number;   // default 500 chars
  overlap: number;        // default 100 chars
  minChunkSize: number;   // default 50 chars — skip tiny trailing chunks
}

interface ContentChunk {
  text: string;
  chunkIndex: number;
  charOffset: number;     // position in original content
}

function chunkContent(content: string, options?: ChunkOptions): ContentChunk[];
```

**QA Pair Format**:
```typescript
interface QAPair {
  pairId: string;                // unique ID: `${conversationId}:${startSequenceNumber}`
  conversationId: string;
  startSequenceNumber: number;
  endSequenceNumber: number;
  pairType: 'conversation_turn' | 'trace_pair';
  sourceId: string;              // messageId or traceId
  question: string;              // full user message / tool invocation (before chunking)
  answer: string;                // full assistant response / tool result (before chunking)
  contentHash: string;           // hash of question + answer
  workspaceId?: string;
  sessionId?: string;
}
```

**Embedding Metadata** (per chunk in conversation_embedding_metadata):
```typescript
{
  pairId: string;                // links all chunks of the same QA pair
  side: 'question' | 'answer';  // which part of the pair this chunk is from
  chunkIndex: number;            // position within that side's chunks
  conversationId: string;
  startSequenceNumber: number;
  endSequenceNumber: number;
  pairType: 'conversation_turn' | 'trace_pair';
  workspaceId?: string;
  sessionId?: string;
  contentHash: string;
  contentPreview: string;        // first ~200 chars of this chunk for display
}
```

**Conversation Search Result** (returned to caller — always full pair):
```typescript
interface ConversationSearchResult {
  conversationId: string;
  conversationTitle: string;
  sessionId?: string;
  workspaceId?: string;
  pairId: string;
  matchedSequenceRange: [number, number];
  question: string;              // FULL user message / tool invocation
  answer: string;                // FULL assistant response / tool result
  matchedSide: 'question' | 'answer';  // which side the best chunk matched on
  distance: number;                    // raw L2 distance (before reranking)
  score: number;                       // reranked score (lower = better, after boosts applied)
  pairType: 'conversation_turn' | 'trace_pair';
  // Only populated in Scoped mode:
  windowMessages?: MessageData[];
}
```

**SearchMemory Enhanced Parameters**:
```typescript
interface SearchMemoryParams {
  query: string;
  workspaceId: string;           // required
  sessionId?: string;            // optional — triggers Scoped mode
  memoryTypes?: ('traces' | 'states' | 'conversations')[]; // add 'conversations'
  searchMethod?: 'semantic' | 'exact' | 'mixed';
  windowSize?: number;           // default 3, only used in Scoped mode
  dateRange?: { start?: string; end?: string };
  limit?: number;                // default 20, max 100
}
```
