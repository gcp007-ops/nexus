## Summary

Add three new built-in app agents for macOS-local productivity data:

- `apple-calendar` for Apple Calendar data already synced on the machine
- `apple-mail` for Apple Mail data already synced on the machine
- `outlook-local` for Outlook for Mac data, explicitly marked experimental

These apps do not introduce OAuth flows inside Nexus. Instead, they reuse accounts already connected in the local desktop apps and access data through a shared macOS automation layer. The initial implementation is macOS desktop only, uses `BaseAppAgent` with `credentials: []`, exposes a manual validation action in settings, and keeps Outlook for Mac read-only until its script surface is proven reliable.

The plan is intentionally scoped to the current plugin architecture:

- App agents remain discoverable through the existing `getTools` / `useTools` two-tool flow
- All platform-specific behavior is runtime-gated inside the app agents and tools
- The plugin manifest remains `isDesktopOnly: false`
- No native helper binary is required in v1

## External Facts That Shape The Plan

- Gmail and Google Calendar do not offer a strong local desktop data API for this plugin shape; their desktop story is mainly browser/PWA/offline cache, not an embeddable local app contract.
- Apple Mail is explicitly automatable on macOS via Script Editor / Automator / Shortcuts.
- Apple Calendar is the strongest macOS calendar target because accounts sync locally and Apple exposes EventKit officially, even if Nexus does not use EventKit directly in v1.
- Outlook for Mac is the weakest of the three targets. Microsoft has removed injection-based plugin support in newer Outlook for Mac builds, so a local Nexus integration should rely only on app automation and should be treated as version-sensitive and experimental.

This is why the plan targets Apple Mail and Apple Calendar as first-class apps, while `outlook-local` is intentionally narrower in phase 1.

## Scope

### In Scope

- macOS-only local app integrations for:
  - Apple Mail
  - Apple Calendar
  - Outlook for Mac
- New built-in app agents registered through `AppManager`
- A shared macOS automation layer for spawning `osascript`, passing structured inputs, and parsing structured JSON output
- Manual validation flows in app settings
- Read-focused tools first, with limited write operations only where the local automation path is stable

- Gmail direct local integration

- Browser scraping, Accessibility UI scraping, or DOM scraping of running apps
- Bundling a native Swift helper or shipping a signed companion app in v1
- Mobile support
- Sending mail or mutating Outlook data in v1 if AppleScript support is inconsistent

## Recommended Product Shape

### App Split

Implement three separate installable built-in apps, not one combined "mail/calendar" app.

| App ID | Agent Name | Purpose | Initial Status |
|--------|------------|---------|----------------|
| `apple-calendar` | `appleCalendar` | Read and create events from local Apple Calendar data | First-class |
| `apple-mail` | `appleMail` | Read/search local Apple Mail data and create drafts | First-class |
| `outlook-local` | `outlookLocal` | Read Outlook for Mac data when scriptable | Experimental |

This split is preferable because:

- install/uninstall stays simple in the current app registry
- tool discovery remains clean for LLMs
- Apple app reliability is higher than Outlook for Mac and should not be coupled
- each app can expose its own validation result and docs URL

### Shared Integration Layer

Do not duplicate process execution logic inside every tool. Create one shared macOS automation layer responsible for:

- desktop/macOS platform guards
- locating and spawning `/usr/bin/osascript`
- passing arguments without shell interpolation
- returning parsed JSON or structured execution errors
- mapping permission failures into actionable messages

### Scripting Strategy

Use JavaScript for Automation via `osascript -l JavaScript` in v1.

Rationale:

- JSON serialization is much cleaner than raw AppleScript text parsing
- script inputs can be passed as argv payloads with less quoting risk
- one bridge format can support all three apps
- it avoids committing to a native helper before the product proves useful

Fallback: if a specific Outlook scenario cannot be represented cleanly in JXA, allow a targeted AppleScript implementation for that app only. Do not introduce UI scripting.

#### Design Approach

Three app agents share a common `MacAutomationRunner` and app-specific script builders. Each tool:

1. validates platform and app availability
2. constructs a structured request payload
3. runs a JXA script through the shared runner
4. parses a versioned JSON envelope
5. converts that envelope into `prepareResult(...)`

The JSON envelope should be standardized across all local apps:

```typescript
interface MacAutomationSuccess<T> {
  success: true;
  data: T;
}

interface MacAutomationFailure {
  success: false;
  error: string;
  code:
    | 'NOT_DESKTOP'
    | 'NOT_MACOS'
    | 'APP_NOT_FOUND'
    | 'AUTOMATION_DENIED'
    | 'SCRIPT_ERROR'
    | 'UNSUPPORTED_OPERATION'
    | 'UNEXPECTED_OUTPUT';
}
```

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| App packaging | One combined macOS app vs three separate apps | Three separate apps | Better install granularity, cleaner discovery, Outlook isolation |
| Runtime bridge | AppleScript text vs JXA vs native helper | JXA first | Better JSON output, no native binary packaging |
| Validation model | Credentials-based vs manual capability check | Manual capability check | These apps have no OAuth/API keys |
| Platform gating | `manifest.isDesktopOnly` vs runtime check | Runtime check | Preserve existing mobile-capable plugin manifest |
| Outlook scope | Full CRUD vs read-only first | Read-only first | Outlook for Mac automation is less stable |
| Shared layer | Per-app shelling vs shared runner | Shared runner | Consistent error handling, fewer quoting bugs |
| Output format | Raw text parsing vs JSON envelope | JSON envelope | Deterministic parsing and testability |

#### Interface Contracts

```typescript
interface MacAutomationRequest {
  operation: string;
  args?: Record<string, unknown>;
}

interface AppAvailabilityResult {
  installed: boolean;
  automationAccessible: boolean;
  details?: string;
}

interface AppleCalendarListEventsParams {
  calendarNames?: string[];
  start: string; // ISO
  end: string;   // ISO
  includeNotes?: boolean;
}

interface AppleMailSearchMessagesParams {
  mailboxNames?: string[];
  from?: string;
  subject?: string;
  text?: string;
  limit?: number;
}
```

This is the foundation.

#### What To Change

- Add a desktop/macOS guard before any process work
- Spawn `/usr/bin/osascript` directly without a shell

- Capture stdout/stderr and parse a single JSON envelope
