## Summary

Fix issue `#64` by removing large prompt payloads from local CLI argv, making the transport failure visible to users, and adding a tight preflight guard so the adapter fails before Windows process creation does.

This plan is based on the current codebase, not the issue text:

- `AnthropicClaudeCodeAdapter` still passes the system prompt via `--append-system-prompt` and the user prompt as the final positional argv item.
- `GoogleGeminiCliAdapter` still passes the full combined prompt via `--prompt <combinedPrompt>`.
- `SystemPromptBuilder` can already create very large prompt text because it inlines file contents from `contextNotes`.
- `MessageManager` still collapses provider failures to `Failed to send message`.
- `ClaudeHeadlessService` uses the same prompt-as-argv pattern and should be audited as an adjacent risk, even if it is not the primary fix.

This plan is intentionally transport-only. It does not change compaction policy or context budgeting beyond the transport-size guard required to avoid silent local CLI failures. That work belongs in `docs/plans/context-compaction-engineering-plan.md`.

---

## Ground Truth From Code

The current behavior matters because the fix should match the real call paths:

- `AnthropicClaudeCodeAdapter.generateStreamAsync()` uses `spawnDesktopProcess(childProcess, runtime.claudePath, args, ...)` with `stdio: ['ignore', 'pipe', 'pipe']`.
- That adapter currently parses Claude Code `stream-json` output line by line from `stdout` and collects `stderr` separately.
- `GoogleGeminiCliAdapter.generateUncached()` uses `runCliProcess(runtime.geminiPath, args, ...)` and then parses the returned JSON.
- `runCliProcess()` already handles stdout/stderr collection and `child.on('error')`, but it does not support stdin today.
- `LLMProviderError` already carries `provider`, `code`, and `originalError`, so we do not need a new error type.
- `MessageManager.sendMessage()` is the last UI hop before the error is shown, and it currently discards the real message.

## Claude Code Prompt Constraint

The documented Claude Code system-prompt semantics matter here:

- `--append-system-prompt` appends small inline text to the default prompt.
- `--system-prompt-file` replaces the default system prompt with file contents.

The approved path should preserve current Claude Code semantics unless we prove replacement-file parity separately. That means the main fix keeps `--append-system-prompt` for bounded, small prompt text, moves only the user prompt to stdin, and adds a strict preflight guard for oversized appended system-prompt payloads.

`--system-prompt-file` is still useful as a later exploratory path, but only as a follow-up with explicit semantic review. It should not be the primary fix because it changes the prompt contract by replacing Claude Code's default prompt instead of appending to it.

## Implementation Order

### Phase 1: Fix Claude Code first
This is the user-visible bug.

1. Send the user prompt through stdin instead of argv.
2. Keep `--append-system-prompt` only for bounded, small prompt text.
3. Add a strict preflight budget check for the assembled system-prompt payload before spawn.
4. Map spawn-time transport failures to a clear `LLMProviderError`.
5. Keep streaming parsing, tool extraction, and auth checks unchanged.
6. Defer any `--system-prompt-file` experimentation to a separate follow-up path, because it changes semantics.

### Phase 2: Harden Gemini CLI next
This is the same risk class, but the current code path is different.

1. Decide whether Gemini can safely consume stdin through the existing process helper.
2. If yes, extend `runCliProcess()` to support stdin text.
3. If not, keep the current JSON request path but fail fast on oversized prompt transport.
4. Preserve the existing temp system-settings file and usage extraction.

### Phase 3: Audit Claude Headless
This service has the same argv pattern.

1. Review whether the service is still user-facing or just experimental.
2. If it is active, apply the same prompt transport policy.
3. If it is not worth fixing now, document the deferred risk in this plan and in the implementation PR.

## File-By-File Plan

### `src/services/llm/adapters/anthropic-claude-code/AnthropicClaudeCodeAdapter.ts`

This is the primary fix.

#### What to change
- Replace the prompt-bearing argv path with a transport helper that returns:
  - `args` for flags only
  - `stdinText` for the user prompt
  - `estimatedArgvChars` for the preflight check
- Keep `--append-system-prompt` for the current bounded system prompt content.
- Keep model selection, tool flags, auth status, and JSON stream parsing as-is.
- Change `stdio` from `['ignore', 'pipe', 'pipe']` to `['pipe', 'pipe', 'pipe']` so stdin can be written.
- Write the user prompt to `child.stdin` and close stdin explicitly.
- Add `child.on('error')` handling so spawn-time failures do not become silent process crashes.
- Map `ENAMETOOLONG` and `E2BIG` to a clear transport error.
- Keep non-transport runtime failures as `PROVIDER_ERROR`.

#### Pseudocode
```typescript
const transport = buildClaudePromptTransport({ systemPrompt, prompt })
const child = spawnDesktopProcess(...transport.args, { stdio: ['pipe', 'pipe', 'pipe'] })

child.once('error', (error) => reject(mapTransportError(error)))
if (child.stdin && transport.stdinText) {
  child.stdin.end(transport.stdinText)
}

for await (const line of readline.createInterface({ input: child.stdout })) {
  // existing stream-json parsing
}

const closeResult = await closePromise
if (closeResult.exitCode !== 0) throw providerErrorFromStderr(stderr)
```

#### Edge cases
- Windows `.cmd` / `.bat` wrappers still count against command-line length; shell escaping does not remove that limit.
- Large `contextNotes` payloads from `SystemPromptBuilder` are exactly the case this needs to survive.
- `--append-system-prompt` still consumes argv space, so the plan needs a strict preflight limit for oversized system-prompt payloads.
- If stdin is not closed, Claude Code can hang waiting for input.
- Spawn can fail before `close`; that must be caught by the `error` event listener, not only by exit-code logic.
- A later `--system-prompt-file` path would replace Claude Code's default prompt, so it is explicitly follow-up only.

#### Failure modes to preserve
- Missing `claude` binary must still report `CONFIGURATION_ERROR`.
- Missing `node` or `connector.js` must still report `CONFIGURATION_ERROR`.
- Auth failure must still report `AUTHENTICATION_ERROR`.
- Non-zero exit from Claude Code must still include `stderr` and report `PROVIDER_ERROR`.
- Payload overflow must report `REQUEST_TOO_LARGE`.

### `src/ui/chat/services/MessageManager.ts`

This is the user-facing error surfacing fix.

#### What to change
- Replace the hardcoded `Failed to send message` fallback in the catch path with the actual error message.
- Keep abort handling unchanged.
- If you want a fallback, only use it when the error object has no usable message.

#### Expected behavior
- A transport failure should show the real message in the chat UI.
- A provider-specific auth or runtime error should remain visible instead of being flattened to a generic string.

#### Pseudocode
```typescript
catch (error) {
  const wasAborted = await this.abortHandler.handleIfAbortError(error, conversation, aiMessageId)
  if (!wasAborted) {
    this.events.onError(getErrorMessage(error))
  }
}
```

### `src/services/llm/adapters/google-gemini-cli/GoogleGeminiCliAdapter.ts`

This is the sibling transport-risk fix.

#### What to change
- Keep the existing temp `system-settings.json` behavior.
- Decide whether the prompt can be moved off argv.
- If stdin is supported by the helper path, use it for the combined prompt.
- If stdin is not supported cleanly, add a strict preflight transport check and throw before spawn.
- Keep the JSON parsing and usage extraction unchanged.

#### Preferred implementation path
- Extend `runCliProcess()` to optionally accept `stdinText`.
- Feed the combined prompt through stdin when the helper supports it.
- Keep `--model` and `--output-format json` as argv.

#### Fallback implementation path
- Keep the current argv transport only if necessary.
- Add a clear error before spawn when the combined prompt is too large.
- Do not let a raw Windows spawn error leak back to the UI.

#### Pseudocode
```typescript
const combinedPrompt = this.buildPrompt(prompt, options?.systemPrompt)
if (isTooLargeForTransport(combinedPrompt)) {
  throw new LLMProviderError('Gemini CLI prompt is too large for local argv transport.', this.name, 'REQUEST_TOO_LARGE')
}

const result = await runCliProcess(runtime.geminiPath, args, {
  cwd: runtime.vaultPath,
  env: buildGeminiCliEnv(settingsPath, runtime.nodePath),
  stdinText: combinedPrompt
})
```

### `src/utils/cliProcessRunner.ts`

This file is only needed if Gemini gets stdin support through the shared helper.

#### What to change
- Add an optional `stdinText?: string` parameter.
- When `stdinText` is present, spawn with `stdio: ['pipe', 'pipe', 'pipe']` instead of `['ignore', 'pipe', 'pipe']`.
- Write the text to `child.stdin` and close it.
- Keep the current stdout/stderr collection and `child.on('error')` resolution logic.
- Preserve the current behavior when `stdinText` is absent.

#### Tests this requires
- helper resolves successfully when stdin text is written
- helper resolves with `exitCode: null` on spawn error
- helper does not break existing no-stdin callers such as auth status
- helper still leaves `claude auth status` untouched when `stdinText` is omitted

### `src/services/external/ClaudeHeadlessService.ts`

This is the adjacent audit target.

#### What to change
- Review whether the service should receive the same prompt transport fix.
- If yes, apply the same "payload off argv" rule.
- If no, keep the plan explicit that the service remains a known transport risk.

#### Why this matters
- It uses the same final-argv prompt pattern as Claude Code.
- It is not the primary fix for issue `#64`, but it shares the same failure class.

### `src/services/chat/StreamingResponseService.ts`

This file should stay mostly untouched.

#### Only change it if needed
- Verify that provider errors are not being rewritten during streaming completion.
- If no rewrite exists, leave it alone.

## Exit Criteria

- Claude Code no longer sends large prompt text through argv.
- Gemini CLI no longer fails silently on oversized prompt transport.
- The UI shows the real provider error instead of a generic failure string.
- Temp files are cleaned up on success and failure.
- Existing auth, tool parsing, and JSON parsing behavior remains unchanged.
