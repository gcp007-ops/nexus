## Summary

Fix issue `#124` by making provider settings persistence honest and failure-aware. The current provider flow has three related problems:

- `ProvidersTab.ts` fires async saves for secondary OAuth provider config changes inside `void` IIFEs, so persistence failures are unhandled.
- `LLMProviderModal.ts` marks the save as complete before the async write actually finishes, which can show a false success state.
- The secondary OAuth connect/disconnect/status flows in `GenericProviderModal.ts` can surface success notices before persistence is confirmed.

The fix should keep the existing provider UX, but it must not claim success until the write has actually completed.

---

## Ground Truth From Code

The bug is not theoretical; the current code paths match the issue report:

- `src/settings/tabs/ProvidersTab.ts` builds secondary OAuth provider configs for OpenAI/Codex, Claude Code CLI, and Gemini CLI.
- Each secondary provider currently stores the new config and then calls `await this.saveSettings()` inside a fire-and-forget async closure.
- `src/components/LLMProviderModal.ts` treats `onSave` as synchronous, but the actual save path in `ProvidersTab` is async.
- The modal also shows `Saved` immediately after calling `onSave`, not after persistence resolves.
- `src/components/llm-provider/providers/GenericProviderModal.ts` calls the secondary OAuth `onConfigChange` callback from connect/disconnect/status-check flows without awaiting persistence.

This means the UI can report success while the data never reaches plugin storage.

#### Design Approach

Make persistence explicit at the boundary where the UI asks the settings layer to save. The modal and provider tabs should treat save operations as promises, await them, and only then display success. That gives one consistent contract for both the primary provider config and secondary OAuth sub-providers.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Callback contract | `void` vs `Promise<void>` | `Promise<void>` | The UI needs to know when persistence actually finished |
| Success feedback | Immediate vs post-save | Post-save only | Avoid false-positive "Saved" states |
| Failure feedback | Silent vs explicit | Explicit | Users need a visible retry signal when writes fail |
| Save sequencing | Fire-and-forget vs awaited | Awaited | Prevent the UI from outrunning storage writes |
| Scope of fix | Narrow path only vs shared contract | Shared contract | Prevents the same bug from reappearing in adjacent save paths |

#### Interface Contracts

```typescript
interface LLMProviderModalConfig {
  onSave: (config: LLMProviderConfig) => Promise<void>;
}

interface SecondaryOAuthProviderConfig {
  onConfigChange: (config: LLMProviderConfig) => Promise<void>;
}
```

The modal should only show `Saved` after the save promise resolves. If the promise rejects, the modal should show `Save failed` and the caller should decide whether to raise a `Notice`.
