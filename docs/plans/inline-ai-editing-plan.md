# Implementation Plan: Inline AI Editing

> Generated by `/PACT:plan-mode` on 2026-02-05
> Status: IMPLEMENTED

## Summary

Add inline AI editing capability allowing users to select text, trigger via right-click menu or hotkey, enter an instruction, and have an LLM transform the text. Uses a single modal with state machine (INPUT → LOADING → RESULT) pattern, streaming responses, and apply/retry/cancel actions.

#### Design Approach

**State Machine Pattern:**
```
INPUT → LOADING → RESULT
         ↑          |
         └─ Retry ──┘
```

**Component Structure:**
```
src/
├── services/
│   └── InlineEditService.ts        # Business logic, state management
├── ui/
│   └── inline-edit/
│       ├── InlineEditModal.ts      # UI component, renders states
│       ├── components/
│       │   └── ModelSelector.ts    # Model dropdown component
│       └── types.ts                # Feature-specific types
└── main.ts                         # Command/menu registration
```

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Modal architecture | Single modal, Two modals | Single modal with states | Cleaner UX, easier retry flow, maintains context |
| Response display | Complete only, Streaming | Streaming with final display | Better UX, LLMService already supports streaming |
| Concurrent requests | Allow multiple, Block, Queue | Block concurrent | Simple, prevents confusion, revisit if needed |
| State management | Class properties, State machine | Explicit state machine | Predictable transitions, testable, self-documenting |
| Empty selection | Allow, Require selection | Require selection | Prevents accidental whole-note edits, clearer UX |
| Mobile support | Desktop-only, Hotkey-only | Hotkey-only | Context menu unavailable, but command palette works |
| Model persistence | Save last, Always default | Always default | Simpler UX, uses configured chat model |

#### Interface Contracts

```typescript
interface InlineEditRequest {
  selectedText: string;
  instruction: string;
  context?: {
    fileName: string;
    cursorPosition: EditorPosition;
  };
  modelConfig: {
    provider: string;
    model: string;
  };
}

interface InlineEditResult {
  success: boolean;
  editedText?: string;
  error?: string;
  tokenUsage?: { input: number; output: number };
}

type InlineEditState =
  | { phase: 'input'; selectedText: string }
  | { phase: 'loading'; progress?: string }
  | { phase: 'result'; original: string; edited: string }
  | { phase: 'error'; message: string };
```

## Open Questions

### Resolved (User Decisions)
- [x] **Empty selection behavior**: Require selection — show error if nothing selected. Prevents accidental whole-note edits.
- [x] **Mobile support**: Hotkey-only on mobile — no context menu, but command available via hotkey or command palette.
- [x] **Persist model selection**: No — always use configured default chat model. Simpler UX.

### Require Further Research
- [ ] **Editor selection persistence**: Does opening a Modal cause Obsidian's Editor to lose selection? Need to test and document workaround if so.
- [ ] **LLM cancellation**: Does LLMService support AbortController for mid-stream cancellation? Critical for cancel button.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Selection lost when modal opens | Medium | High | Capture selection text AND position before modal opens |
| LLM errors losing user input | Medium | High | Preserve instruction on error, allow retry |
| Streaming cancellation not supported | Medium | Medium | Implement AbortController if missing, or disable cancel during streaming |
| Modal styling doesn't match Obsidian themes | Low | Medium | Use CSS variables exclusively |
| Memory leak from abandoned streams | Medium | Medium | Proper cleanup in modal `onClose()` handler |
