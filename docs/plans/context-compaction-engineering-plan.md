## Summary

Build a provider-agnostic compaction system that keeps long conversations bounded without losing recoverability.

The codebase already has the right building blocks:
- `TokenCalculator` estimates context usage and can consume provider usage when available.
- `ContextCompactionService` compacts old conversation messages into a summary.
- `ContextPreservationService` forces `createState` before truncation.
- `SystemPromptBuilder` injects compaction output into the system prompt.
- `MessageRepository.getMessagesBySequenceRange()` can retrieve exact transcript windows.
- `searchMemory` already supports session-scoped window retrieval.

The gaps are architectural, not foundational:
- auto-compaction is effectively gated to `webllm`
- `previousContext` is session-memory only, not durable conversation state
- compaction output is a single flat summary, not a hierarchical frontier
- exact transcript recovery is only partially modeled through search and sequence ranges
- `createState` is a workspace resume tool, not the compaction store

This plan therefore defines a hierarchical compaction frontier:
- append-only raw transcript remains the source of truth in JSONL
- compaction records are durable conversation metadata
- the active frontier is rendered into the system prompt as a stable prefix
- old compaction records can be compacted again into meta-records
- exact transcript references are sequence-range based, with optional anchor message IDs

This plan is separate from local CLI transport hardening. That work is covered by `docs/plans/claude-code-connect-silent-failure-plan.md`.

## What The Code Does Today

### Prompt Assembly
- `SystemPromptBuilder` already places `<previous_context>` near the top of the system prompt.
- For most non-Google providers, `ProviderMessageBuilder` already flattens conversation history into the system prompt text instead of keeping it fully structured.
- Google is the exception: it keeps structured `conversationHistory` in Google format.

### Current Compaction Flow
- `ChatView.handleSendMessage()` calls `shouldCompactBeforeSending(message)` before send.
- `ModelAgentManager` currently initializes `ContextTokenTracker` only for `webllm`.
- `ChatView.performContextCompaction()` first tries `ContextPreservationService.forceStateSave()`.
- If `createState` succeeds, the returned content replaces the heuristic summary.
- `ContextCompactionService.compact()` truncates old messages while preserving atomic message units.

### Current Persistence Model
- `ConversationRepository` already persists `metadataJson`, plus denormalized `workspaceId`, `sessionId`, `workflowId`, `runTrigger`, `scheduledFor`, and `runKey`.
- `StateRepository` writes immutable state snapshots to workspace JSONL plus SQLite metadata.
- `createState` currently saves workspace-scoped states using session ID `_workspace`.
- `createState` expects `conversationContext`, `activeTask`, `activeFiles`, `nextSteps`, and optional `tags`.

### Current Retrieval Model
- `MessageRepository` already supports exact message retrieval by `conversationId` and `sequenceNumber` range.
- `ConversationEmbeddingService` and `ConversationWindowRetriever` already work in sequence ranges, not line numbers.
- `ConversationEmbeddingWatcher` already emits `QAPair`s with `conversationId`, `startSequenceNumber`, `endSequenceNumber`, `workspaceId`, and `sessionId`.
- `searchMemory` is strong for semantic discovery and session windows, but not yet for exact transcript-reference recovery.

## Design Rules

### 1. Compaction belongs in the system prompt
The carried-forward compaction frontier should be rendered as system prompt prefix material, not as fake user messages.

Reason:
- it is persistent context, not new user intent
- it is the most stable prefix for caching where providers support prompt caching
- this repo already treats prior context as prefix material for most non-Google providers

The plan should not promise cache savings from role alone. Caching is prefix-based and provider-specific.

### 2. The source of truth stays append-only
Raw conversation messages remain in JSONL. Compaction never deletes the source of truth. It only changes the active prompt projection.

### 3. Workspace states are not compaction state
`createState` remains a workspace resume mechanism. Compaction records are conversation metadata. The two can reference each other, but they are not the same store.

### 4. Exact transcript references use sequence ranges
Do not use line numbers. Use:
- `conversationId`
- `sessionId`
- `startSequenceNumber`
- `endSequenceNumber`
- optional `messageIds`
- optional anchor excerpts

### 5. Token estimates are still needed
Provider usage is only available after a request completes, and some providers never return exact usage. Pre-send budgeting therefore still needs the rough fallback estimate.

For rollout, do not block the first compaction PR on a new tokenizer dependency. PR 1 should normalize usage accounting and keep the current rough `chars / 4` fallback. A later phase can introduce tokenizer-backed estimation once the abstraction and library choice are confirmed.

### 6. Frontier growth needs recursive compaction
Once the frontier itself becomes large, compact the oldest frontier records into meta-records. Do not let summaries become the next overflow source.

## Execution Model

### Runtime Flow
1. Load the conversation and restore any persisted compaction frontier from conversation metadata.
2. Before send, compute a budget estimate using the existing rough fallback plus any provider-reported usage already known.
3. If the conversation is approaching the safe threshold, warn.
4. If it crosses the compaction threshold, run preservation first, then compact old messages into a new frontier record.
5. Persist the new frontier record(s) into conversation metadata.
6. Render the frontier into the system prompt on the next send.
7. If the frontier itself grows too large, meta-compact the oldest records into a higher-level record.

### Pseudocode
```ts
budget = estimateConversationBudget(conversation, systemPrompt, providerPolicy)

if budget.action === 'warn':
  showWarning()

if budget.action === 'compact':
  preservedState = tryCreateState(conversation)
  compacted = compactOldMessages(conversation, preservedState)
  persistCompactionRecord(conversationId, compacted)
  updateFrontierInConversationMetadata(conversationId, compacted.frontier)

if frontierTokens > frontierBudget:
  metaCompacted = compactFrontierRecords(frontier)
  persistCompactionRecord(conversationId, metaCompacted)
```

```ts
resolveTranscript(ref):
  if ref.messageIds exist:
    fetch by id and sort by sequenceNumber
  else:
    fetch messages by conversationId + sequence range
  return exact transcript slice
```

## Step-By-Step Plan

### Phase 1: Normalize budgeting and usage accounting
Goal: make compaction decisions based on a consistent estimate, not a single provider-specific tracker.

1. Add a shared `ContextBudgetService`.
2. Move the current provider-specific compaction gate out of `ModelAgentManager` and into policy-driven budgeting.
3. Normalize usage shape in `StreamingResponseService` and `TokenCalculator`.
4. Keep provider-reported usage as reconciliation data, not the only source of truth.
5. Keep the existing rough estimation path for pre-send budgeting in this phase.

`Provider budget policy` for the first rollout:
- `webllm` remains special because it has a hard 4K limit
- `claude-code`, `gemini-cli`, `openai-codex`, and `github-copilot` get a conservative 200k soft cap
- model metadata remains unchanged; the soft cap is an operating policy

Tokenizer-backed estimation is a later phase. The plan does not require choosing a tokenizer dependency before PR 1 ships.

### Phase 2: Make compaction durable
Goal: store frontier records in conversation metadata and restore them on load.

1. Extend conversation metadata with a `compaction` object.
2. Persist the active frontier after every compaction.
3. Restore the frontier in `ModelAgentManager.initializeFromConversation()`.
4. Replace the single in-memory `previousContext` with frontier state loaded from metadata.
5. Keep `previousContext` only as a compatibility shim while the migration lands.

### Phase 3: Make system prompt projection explicit
Goal: render the active frontier in a stable, cache-friendly prompt prefix.

1. Update `SystemPromptBuilder` to render a structured frontier section first.
2. Keep the active frontier deterministic: same records, same order, same formatting.
3. Avoid duplicating frontier content in both system prompt and message history.
4. Preserve the latest live user turn as the live turn, but keep the carried-forward conversation context as prefix material.

### Phase 4: Add exact transcript recovery
Goal: let compaction records point back to the exact transcript slice that was collapsed.

1. Extend compaction records with sequence-range coverage metadata.
2. Extend `createState` payloads to include transcript coverage refs as additive metadata.
3. Add an internal transcript recovery helper/service first, built directly on `MessageRepository.getMessagesBySequenceRange()`.
4. Have `searchMemory` optionally wrap that helper later if exact-reference retrieval should become user-facing there.
5. Reuse `MessageRepository.getMessagesBySequenceRange()` for exact recovery.

### Phase 5: Add hierarchical meta-compaction
Goal: keep the frontier bounded even after many compactions.

1. Define a frontier token budget separate from raw conversation budget.
2. When frontier usage crosses its threshold, compact the oldest frontier records into a meta-record.
3. Keep only the active frontier in the prompt projection.
4. Preserve ancestry references so older context can still be recovered via metadata or search.

- Run the shared budget check before every send.
- Trigger preservation and compaction when the action is `compact`.
- Persist the updated frontier after compaction.
- Reset token accounting only after the frontier is updated.
- Leave the UI notice flow intact, but make it clear whether the action was preserve, compact, or meta-compact.

### `src/ui/chat/services/SystemPromptBuilder.ts`
- Replace the single `<previous_context>` block with a structured frontier block.
- Put the frontier before other dynamic context sections so it becomes part of the stable prefix.
- Include coverage refs in the prompt in a compact, machine-readable form.
- Keep the section ordering deterministic.

### `src/services/llm/core/ProviderMessageBuilder.ts`
- Keep carried-forward context in the prompt prefix instead of duplicating it as synthetic user/assistant turns.
- For non-Google providers, stop producing two separate historical representations for the same frontier.
- For Google, preserve the provider’s required structured history format, but still feed the frontier through the same prompt-prefix semantics.

### `src/services/chat/ContextCompactionService.ts`
- Change the return shape from a flat `CompactedContext` toward a durable compaction record.
- Capture the removed transcript coverage using `conversationId` and sequence numbers.
- Preserve atomic message units as the current implementation already does.
- Keep the existing summary/file/topic heuristics, but attach ancestry and coverage.

### `src/services/chat/ContextPreservationService.ts`
- Keep the pre-compaction `createState` call.
- Teach the save-state prompt to capture the exact work context plus transcript anchors.
- Pass through sequence-range coverage when available.
- Do not make `createState` responsible for conversation compaction storage.

### `src/ui/chat/utils/TokenCalculator.ts`
- Normalize both snake_case and camelCase usage payloads.
- Keep the text-length fallback for opaque providers.
- Keep the current rough estimate for pre-send decisions in this phase.
- Reconcile estimates with real provider usage after send when available.

### `src/services/chat/StreamingResponseService.ts`
- Persist usage in one canonical internal shape.
- Preserve compatibility when older messages already have the legacy shape.
- Do not overwrite a more accurate usage record with an empty one.

### `src/database/repositories/ConversationRepository.ts`
- Store the compaction frontier in `metadataJson`.
- Preserve existing `workspaceId` and `sessionId` denormalization.
- Make it possible to restore the active frontier on conversation reload.

### `src/agents/memoryManager/tools/states/createState.ts`
- Keep the existing required fields intact.
- Add optional transcript coverage metadata to the saved state payload.
- Keep the current workspace-scoped state behavior and `_workspace` session ID.

### `src/agents/memoryManager/types.ts`
- Extend `CreateStateParams` with optional transcript reference fields.
- Keep existing required fields so current callers continue to work.

### `src/agents/searchManager/tools/searchMemory.ts`
- Keep semantic search as the discovery path.
- Optionally wrap the internal transcript recovery helper later if exact-reference retrieval should be exposed here.
- Surface sequence-range coverage in results when available.

### `src/services/chat/ConversationTranscriptRecoveryService.ts`
- Add the first exact transcript recovery implementation here, built on `MessageRepository.getMessagesBySequenceRange()`.
- Resolve compaction coverage refs into ordered message slices.
- Keep this service internal to the compaction/search pipeline; do not expose it as a new user-facing tool unless later work decides to wrap it.

### `src/services/embeddings/ConversationWindowRetriever.ts`
- Reuse the existing window retrieval logic for exact transcript windows.
- Keep the window-size mode for semantic recovery, but do not confuse it with exact transcript reference mode.

### `src/database/repositories/MessageRepository.ts`
- No schema change is required for exact transcript retrieval.
- The existing `getMessagesBySequenceRange()` method is the primitive to build on.

## Concrete Data Shapes

### Compaction Record
```ts
interface CompactionRecord {
  id: string;
  level: number;
  summary: string;
  parentIds: string[];
  coverage: Array<{
    conversationId: string;
    sessionId?: string;
    startSequenceNumber: number;
    endSequenceNumber: number;
    messageIds?: string[];
    anchorExcerpts?: string[];
  }>;
  stateId?: string;
  estimatedTokens: number;
  createdAt: number;
  provider?: string;
  model?: string;
}
```

### Conversation Metadata
```ts
metadata.compaction = {
  frontier: CompactionRecord[],
  lastCompactedAt: number,
  lastCompactedSequenceNumber: number,
  provider?: string,
  model?: string
}
```

```ts
type TranscriptCoverageRef = {
  conversationId: string;
  sessionId?: string;
  startSequenceNumber: number;
  endSequenceNumber: number;
  messageIds?: string[];
  anchorExcerpts?: string[];
}
```

## Edge Cases

- Empty conversations should return no compaction record and no frontier update.
- Tool-call chains must be treated as atomic units, not split across compaction boundaries.
- Branch conversations should not inherit a parent frontier blindly.
- Reloads must restore frontier state from metadata, not from transient memory.
- Provider usage may be missing or partially reported; budgeting must still work.
- Exact transcript recovery must fall back cleanly when message IDs are absent and sequence ranges are the only stable anchor.
- Meta-compaction must not preserve both a meta-record and all of its parents in the active frontier.
- A preserved `createState` result can fail verification and still require a fallback to heuristic compaction.
- Compaction should only run on completed turns, not on streaming partial messages.

## Test Plan

### Unit Tests
- `ContextBudgetService` computes `send`, `warn`, `compact`, and `block` correctly.
- `TokenCalculator` reads both camelCase and snake_case usage payloads.
- `ContextCompactionService` preserves atomic units and produces coverage metadata.
- `SystemPromptBuilder` renders the frontier deterministically.
- `ConversationTranscriptRecoveryService` exact recovery returns the correct transcript slice.
- `ConversationRepository` persists and restores the compaction metadata payload.

### Integration Tests
- Long conversation compacts on the current provider cohort.
- A second compaction run meta-compacts the oldest frontier records.
- Reloading the conversation restores the frontier and produces the same prompt prefix.
- `createState` still works as a workspace resume tool during compaction.
- Exact transcript recovery returns the same message range that was collapsed into the compaction record.

### Manual Tests
- Run a long Claude Code conversation and verify the prompt stays bounded after compaction.
- Run a long Gemini CLI conversation and verify the same policy applies.
- Load a conversation after reload and confirm frontier restoration.
- Query an earlier transcript slice through the exact recovery helper and confirm the returned messages match the original sequence range.
