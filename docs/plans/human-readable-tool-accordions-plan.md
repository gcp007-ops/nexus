## Summary

Replace raw `getTools` / `useTools` accordion labels with user-facing action text, and make multi-call `useTools` batches read naturally in both serial and parallel execution. The UI should say things like `Checking available tools`, `Reading Daily note.md`, and `Loaded workspace Client Redesign`, rather than exposing transport-layer tool names. This requires three coordinated changes:

1. Normalize tool calls into a shared UI display model so live streaming and persisted message rendering use the same logic.
2. Teach the accordion to render batch-aware summaries and tense-aware labels.
3. Refactor `useTools` execution so child-call progress is observable; otherwise serial/parallel accordions cannot show accurate in-flight status.

## Specialist Perspectives

### Preparation Phase
**Effort**: Medium

#### Current Code Findings

- `ProgressiveToolAccordion` already supports multiple internal tool rows, but it only stores a single static `name` per row and derives header text from those names. It cannot currently switch between present/past/failure copy or reason about execution strategy.
- The persisted render path in `src/ui/chat/components/factories/ToolBubbleFactory.ts` unwraps wrapper calls, but the live streaming path (`ToolEventCoordinator` -> `MessageBubble` -> `ProgressiveToolAccordion`) does not. This guarantees drift between live and reloaded messages.
- `ToolBubbleFactory.unwrapUseTool()` only checks for `useTool`, while the real tool slug is `useTools`. That mismatch must be fixed or persisted batch rendering will miss real wrapper calls.
- `DirectToolExecutor.executeToolCalls()` emits `started` / `completed` only for the outer tool call. For `useTools`, that means the UI never sees child-call progress today.
- `UseToolTool.execute()` contains the serial/parallel execution logic internally, so there is no current hook for child call lifecycle events without refactoring.
- `StreamingResponseService` persists raw `chunk.toolCalls` onto the assistant message. That is good for compatibility, but it means human-readable labels must be derived at render time, not stored as canonical data.

#### Design Approach

**1. Shared display model**

Introduce a UI-only normalized model so both live and persisted paths operate on the same data shape.

```typescript
interface ToolDisplayGroup {
  id: string;                    // outer tool call id
  kind: 'discovery' | 'single' | 'batch' | 'reasoning';
  technicalName?: string;        // outer raw tool name
  strategy?: 'serial' | 'parallel';
  status: 'pending' | 'queued' | 'executing' | 'completed' | 'failed';
  steps: ToolDisplayStep[];
}

interface ToolDisplayStep {
  id: string;                    // stable child id (outer id or outer id + index)
  technicalName?: string;        // contentManager.read, memoryManager.loadWorkspace, etc.
  parameters?: unknown;
  result?: unknown;
  error?: string;
  status: 'pending' | 'queued' | 'executing' | 'completed' | 'failed' | 'skipped';
}
```

This model is render-only. It does not change stored conversation schemas.

**2. Generic discovery copy for `getTools`**

- In-flight: `Checking available tools`
- Completed: `Checked available tools`
- Failed: `Failed to check available tools`

The main label should not expose schema or agent list text. Raw parameters can remain in the expanded details section if needed.

**3. Tool-specific action formatter**

Create a formatter that maps `(technicalName, parameters, result, status)` to user-facing text.

Examples:

- `contentManager.read` + `path` -> `Reading Daily note.md` / `Read Daily note.md`
- `memoryManager.loadWorkspace` + `id` -> `Loading workspace Client Redesign` / `Loaded workspace Client Redesign`
- `searchManager.searchContent` + `query` -> `Searching notes for "OAuth"` / `Searched notes for "OAuth"`
- `storageManager.move` + `sourcePath` + `destinationPath` -> `Moving draft.md to Archive/` / `Moved draft.md to Archive/`

Unknown tools fall back to the current title-cased action name.

**4. Batch-first rendering for `useTools`**

One outer `useTools` call should render as one accordion with N child rows, not N unrelated accordions.

- `strategy: serial`
  - Header while running: current active action + queued count
  - Example: `Reading Ideas.md, 2 more queued`
  - Header after success: `Completed 3 actions`
  - Header after stop-on-error: `Failed after 1 of 3 actions`

- `strategy: parallel`
  - Header while running: summarize up to 2 active actions, then `+N more`
  - Example: `Reading Ideas.md and loading Client Redesign`
  - Example: `Reading Ideas.md, loading Client Redesign, +2 more`
  - Header after completion: `Completed 4 actions`
  - Mixed result: `Completed 3 actions, 1 failed`

**5. Real child-call progress**

The current code cannot show truthful serial/parallel progress because only outer wrapper events are emitted. Fix this by extracting execution into a reusable service with optional observer hooks.

```typescript
interface ToolBatchObserver {
  onBatchStarted?(batch: BatchStartEvent): void;
  onStepStarted?(step: BatchStepStartEvent): void;
  onStepCompleted?(step: BatchStepCompleteEvent): void;
  onBatchCompleted?(batch: BatchCompleteEvent): void;
}
```

`UseToolTool.execute()` should call the shared execution service without an observer.

`DirectToolExecutor.handleUseTool()` should call the same service with an observer so the chat UI can update the active accordion accurately during execution.

This avoids changing the `ITool` interface while keeping one source of truth for serial/parallel behavior.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Display primitive for `useTools` | Flatten inner calls vs batch accordion | Batch accordion | Preserves strategy, ordering, and child relationships |
| Human-readable labels source | Persist pre-rendered copy vs derive at render time | Derive at render time | Avoids schema churn and stale labels when rules evolve |
| `getTools` copy | Literal tool name vs generic discovery text | Generic discovery text | Infrastructure step, not user-facing action |
| Child progress hook | Leave outer-only events vs refactor execution core | Refactor execution core | Required for truthful serial/parallel status |
| Unknown tools | Hide vs fallback label | Fallback label | Prevents blank or broken accordions during rollout |
| Phase 1 coverage | All tools vs top-value tools first | Top-value first | Faster delivery, lower regression risk |

## Synthesized Implementation Roadmap

### Wave 1
Ship render-time normalization and formatter coverage for `getTools`, `contentManager.read`, and `memoryManager.loadWorkspace`. This gets the core UX language right without waiting on execution refactors.

### Wave 2
Convert `useTools` from flattened child accordions to grouped batch accordions in both live and persisted rendering. This is the minimum structural change needed before strategy-aware copy can work.

### Wave 3
Extract shared batch execution and emit real child-step progress from the native chat executor. This unlocks truthful serial/parallel live updates instead of batch-level guesswork.

### Wave 4
Expand formatter coverage based on observed usage, then polish copy and failure states.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Live UI and persisted UI diverge again | Confusing inconsistent labels | Use one shared normalizer for both code paths |
| `useTools` execution refactor duplicates validation logic | Behavioral drift | Extract execution core into a shared service, keep one validation path |
| Batch grouping breaks existing accordion assumptions | Rendering regressions | Roll out with unit coverage around one-group/one-accordion ownership |
| Unknown tools render poorly during rollout | UX inconsistency | Keep fallback formatter and expand mapping incrementally |
| Provider-executed tools behave differently from native execution | Missing states | Keep normalization logic tolerant of pre-completed tool calls and provider-executed results |

---

## Recommendation

Implement this as a two-layer refactor, not a copy-only patch. The human-readable text is straightforward; the real complexity is making `useTools` render as a batch with accurate live progress. If that execution refactor is skipped, the UI can be prettier but it will still be structurally misleading for serial and parallel runs.
