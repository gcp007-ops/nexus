## Summary

Add extensible OAuth 2.0 PKCE "Connect" buttons to the LLM provider settings UI. Users can authenticate with providers by clicking a button and signing in via their browser, instead of manually copying API keys. **OpenRouter** is the first provider (official, documented). **OpenAI Codex** is included as an experimental feature with a consent dialog. The architecture uses an `IOAuthProvider` interface + `OAuthService` registry so future providers require only a small new file.

#### Design Approach

Extensible registry pattern: `OAuthService` is a singleton that holds a `Map<string, IOAuthProvider>`. Any provider implementing `IOAuthProvider` and registered at startup gets a Connect button in the UI automatically. The settings UI only checks `OAuthService.hasProvider(providerId)` — no hardcoded provider names in the UI layer.

#### Key Decisions

| Decision | Options | Resolution | Rationale |
|----------|---------|------------|-----------|
| OAuthService pattern | Singleton vs factory | **Singleton** | One OAuth flow at a time; simplest state management |
| OAuth state storage | Separate registry vs field on LLMProviderConfig | **`oauth?` on LLMProviderConfig** | Keeps config cohesive; additive/backward-compatible |
| Callback server | Share MCP Express app vs standalone ephemeral | **Standalone ephemeral** | Lifecycle mismatch (MCP = long-lived, callback = seconds); avoids coupling |
| Port strategy | One port for all vs per-provider | **Per-provider ports** | Codex = 1455, OpenRouter = 3000; no cross-provider conflict |
| OpenRouter port 3000 conflict | Route to Express / fallback / error | **EADDRINUSE → user error** (v1) | Rare in practice (most users use IPC); clear message is sufficient |
| PKCE utils | Class methods vs standalone pure functions | **Standalone exports** | Required for testability (test engineer) |
| state parameter | Optional vs mandatory | **Mandatory** | CSRF protection; include even if provider doesn't document it |
| Codex token storage | Separate file vs data.json | **data.json** (OAuthState fields) | Consistent with existing API key storage; warn in consent dialog |
| Codex token refresh | 401-intercept vs proactive | **Proactive** (OpenCode pattern) | Check `expiresAt < Date.now()` before each inference call |
| Codex client ID | Register own vs reuse public | **Reuse** `app_EMoamEEZ73f0CkXaXp7hrann` | No registration process exists; same approach as Cline/OpenCode |
| Codex adapter | Modify OpenAIAdapter vs new adapter | **New `OpenAICodexAdapter`** | Different endpoint (`chatgpt.com/backend-api/codex/responses`), different auth headers |

#### Interface Contracts

```typescript
// src/services/oauth/IOAuthProvider.ts

export interface OAuthProviderConfig {
  providerId: string;          // matches SupportedProvider enum
  displayName: string;         // UI label: "OpenRouter", "ChatGPT (Experimental)"
  authUrl: string;             // Authorization endpoint
  tokenUrl: string;            // Token exchange endpoint
  preferredPort: number;       // OAuth callback port (3000 for OpenRouter, 1455 for Codex)
  callbackPath: string;        // e.g., '/callback' or '/auth/callback'
  scopes: string[];            // OAuth scopes (empty for OpenRouter)
  tokenType: 'permanent-key' | 'expiring-token';
  clientId: string;            // Empty string if none required (OpenRouter)
  experimental?: boolean;      // If true: show consent dialog before flow
  experimentalWarning?: string; // Warning text for consent dialog
}

export interface OAuthResult {
  apiKey: string;              // The key/token to store in apiKey field
  refreshToken?: string;       // For expiring-token providers only
  expiresAt?: number;          // Unix ms, for expiring-token providers only
  metadata?: Record<string, string>; // e.g., { chatgptAccountId }
}

export interface IOAuthProvider {
  readonly config: OAuthProviderConfig;
  buildAuthUrl(callbackUrl: string, codeChallenge: string, state: string): string;
  exchangeCode(code: string, codeVerifier: string, callbackUrl: string): Promise<OAuthResult>;
  refreshToken?(refreshToken: string): Promise<OAuthResult | null>;
}

// State persisted on LLMProviderConfig
export interface OAuthState {
  connected: boolean;
  providerId: string;
  connectedAt: number;         // Unix ms
  refreshToken?: string;
  expiresAt?: number;
  metadata?: Record<string, string>;
}
```

```typescript
// src/services/oauth/OAuthService.ts (simplified API)

export class OAuthService {
  static getInstance(): OAuthService;
  registerProvider(provider: IOAuthProvider): void;
  hasProvider(providerId: string): boolean;
  getProviderConfig(providerId: string): OAuthProviderConfig | null;

  // startFlow: generates PKCE, opens browser, spins up callback server, exchanges code
  // preAuthParams: key_label + credit_limit (for OpenRouter pre-auth form)
  startFlow(providerId: string, preAuthParams?: Record<string, string>): Promise<OAuthResult>;
  cancelFlow(): void;
  refreshToken(providerId: string, refreshToken: string): Promise<OAuthResult | null>;
}
```

#### Implementation Sequence

1. Core types: `IOAuthProvider.ts` (interfaces + OAuthState) + extend `ProviderTypes.ts`
2. Crypto utilities: `PKCEUtils.ts` (standalone pure exports, `crypto.getRandomValues` + `crypto.subtle`)
3. Callback server: `OAuthCallbackServer.ts` (127.0.0.1, single-use, configurable timeout)
4. Service: `OAuthService.ts` (singleton, state machine: idle → authorizing → exchanging → idle)
5. OpenRouter provider: `OpenRouterOAuthProvider.ts` (first working flow end-to-end)
6. UI integration: `GenericProviderModal.ts` + `ProviderModalConfig` extension
7. Wiring: `ProvidersTab.ts` + `main.ts` registration
8. Codex provider: `OpenAICodexOAuthProvider.ts` (experimental, consent dialog, token refresh)

#### Security Requirements (from security-engineer)

- `crypto.getRandomValues()` ONLY — never `Math.random()` for PKCE or state
- `code_challenge_method: 'S256'` always — never `plain`
- `state` parameter mandatory; validate exact match before processing callback
- Bind callback server to `127.0.0.1` (not `localhost` string, not `0.0.0.0`)
- Single-use callback server — shutdown after first valid callback
- 5-minute timeout on callback server
- `code_verifier` in memory only — never persisted, never logged
- Response page: static HTML "You can close this tab" — no dynamic content
- On EADDRINUSE for port 3000: show clear user error, do not silently fail
- OpenAI Codex: consent dialog required before flow begins; warn about data.json token storage
