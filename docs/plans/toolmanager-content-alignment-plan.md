# Implementation Plan: ToolManager Content Transport and executePrompts Alignment

> Generated by GitHub Copilot on 2026-04-20
> Status: APPROVED

## Summary

Ship a focused alignment pass across ToolManager and PromptManager so text-editing flows are reliable and consistent with the current ContentManager surface.

This bundle has three goals:

1. Fix escaped-Unicode handling in CLI-style `useTools` parsing so quoted content like `"\u2014"` becomes real Unicode instead of being written literally.
2. Promote the existing structured `calls` execution path as a first-class `useTools` transport so large note content is not forced through the CLI parser when it does not need to be.
3. Grow `executePrompts` action schema so `replace` can target the current `contentManager.replace` contract instead of relying on the removed `contentManager.update` tool.

This plan explicitly keeps `contentManager.update` removed. All content mutations must route through `write`, `insert`, `replace`, or `setProperty`.

## Problem Statement

Current behavior has two real mismatches:

- `ToolCliNormalizer.unescapeQuotedContent()` decodes common escapes (`\n`, `\t`, quotes, backslash) but not `\uXXXX`, so CLI-transported note content can be silently mangled.
- `executePrompts` still models content actions around the old `update` surface:
  - `append` and `prepend` call `contentManager.update`
  - line-based `replace` also calls `contentManager.update`
  - the action schema only has `position`, which is insufficient for the modern `contentManager.replace` contract (`oldContent`, `newContent`, `startLine`, `endLine`)

There is also a product-shape mismatch:

- internal code already supports structured `useTools.parameters.calls[]`
- public `useTools` schema still pushes models toward the CLI-string `tool` field even for long-form content payloads

## Scope

### In scope

- Escaped-Unicode decoding for quoted CLI values
- `useTools` schema and messaging updates so structured `calls` is clearly supported
- `executePrompts` action-schema expansion for modern replace semantics
- Replacing stale runtime references to `contentManager.update`
- Unit coverage for parser behavior and action validation/execution

### Out of scope

- Reintroducing `contentManager.update`
- Rewriting PromptManager action semantics beyond content actions
- Broad redesign of the ToolManager CLI surface
- Changing `contentManager.replace` validation rules

## Design Decisions

### 1. Keep dual `useTools` transports, but make structured calls explicit

`useTools` should accept either:

- `tool`: CLI-style command string
- `calls`: structured array of `{ agent, tool, params }`

Normalization should continue to prefer `calls` when present. The public parameter schema and tool description should reflect that reality so models and humans are not taught a CLI-only fiction.

### 2. Grow executePrompts replace schema to match real replace semantics

The `replace` action should support two modes:

- **Whole-file replace**
  - existing behavior
  - uses `contentManager.write` with `overwrite: true`
- **Line-range validated replace**
  - new explicit shape
  - uses `contentManager.replace`
  - requires `oldContent`, `startLine`, and `endLine`
  - writes the LLM output as `newContent`

`position` should be removed from the actionable replace contract in favor of explicit line-range fields. If backward compatibility is needed briefly, parse `position` as a deprecated alias for `startLine = endLine = position`, but only when `oldContent` is also supplied.

### 3. Map append/prepend to insert, not write

- `append` -> `contentManager.insert` with `startLine: -1`
- `prepend` -> `contentManager.insert` with `startLine: 1`

This preserves the current content-tool division of responsibilities.

### 4. Unicode decoding must be narrow and safe

Add support for:

- `\uXXXX`

Optional follow-up if needed later:

- `\u{X...}`

This bundle should only add the 4-hex-digit form because that is the concrete missing case observed in the current parser path.

## Schema Changes

### `useTools`

Public parameter schema should allow either CLI or structured execution:

```ts
type UseToolParams = CommonParameters & {
  tool?: string;
  calls?: ToolCallParams[];
  strategy?: 'serial' | 'parallel';
};
```

Validation rules:

- require at least one of `tool` or `calls`
- reject empty string `tool` when `calls` is absent
- preserve current behavior when both are present: `calls` wins

### `executePrompts` content action

Current:

```ts
interface ContentAction {
  type: 'create' | 'append' | 'prepend' | 'replace' | 'findReplace';
  targetPath: string;
  position?: number;
  findText?: string;
  replaceAll?: boolean;
  caseSensitive?: boolean;
  wholeWord?: boolean;
}
```

Proposed:

```ts
interface ContentAction {
  type: 'create' | 'append' | 'prepend' | 'replace' | 'findReplace';
  targetPath: string;
  findText?: string;
  replaceAll?: boolean;
  caseSensitive?: boolean;
  wholeWord?: boolean;

  // Replace-specific fields
  oldContent?: string;
  startLine?: number;
  endLine?: number;

  // Deprecated compatibility alias
  position?: number;
}
```

Replace validation rules:

- whole-file replace: valid when none of `oldContent`, `startLine`, `endLine`, `position` are provided
- line-range replace: valid when `oldContent`, `startLine`, and `endLine` are all present
- deprecated single-line replace: valid when `position` and `oldContent` are present; normalize to `startLine = endLine = position`
- reject partial line-range shapes

## Implementation Sequence

1. Add characterization tests for the current failures.
   - CLI parser writes literal `\u2014` instead of Unicode em dash.
   - `executePrompts` append/prepend still target removed tool.
   - `replace` validation rejects or mishandles line-range semantics.

2. Fix CLI Unicode decoding.
   - Extend `unescapeQuotedContent()` to translate `\uXXXX`.
   - Keep double-escaped `\\u2014` literal.

3. Expose structured `calls` in `useTools` public schema.
   - Update JSON schema and runtime validation.
   - Update tool description/examples to recommend `calls` for content-heavy payloads.

4. Expand `executePrompts` action type and validation.
   - Add `oldContent`, `startLine`, `endLine`.
   - Normalize deprecated `position` when possible.
   - Fail loudly on incomplete replace shapes.

5. Replace stale runtime tool usage.
   - append/prepend -> `contentManager.insert`
   - validated line-range replace -> `contentManager.replace`
   - full-file replace -> `contentManager.write`

6. Clean stale messaging.
   - update `write.ts` guidance text
   - scan for any remaining live `contentManager.update` references

7. Run targeted tests, then broader validation if the targeted set passes.

## Test Plan

### Unit

`tests/unit/ToolManagerCliSyntax.test.ts`

- quoted `"\u2014"` decodes to em dash
- quoted `"\u00D7"` decodes to multiplication sign
- double-escaped `"\\u2014"` stays literal
- raw Unicode input still passes unchanged

`executePrompts` tests

- append action calls `contentManager.insert` with `startLine: -1`
- prepend action calls `contentManager.insert` with `startLine: 1`
- whole-file replace calls `contentManager.write` with `overwrite: true`
- line-range replace calls `contentManager.replace` with `oldContent`, `newContent`, `startLine`, `endLine`
- deprecated `position + oldContent` normalizes to single-line replace
- partial line-range replace is rejected with clear validation error

### Integration / smoke

- `useTools` structured `calls` writes markdown containing em dashes and smart punctuation intact
- `executePrompts` can append to a note, prepend to a note, and replace a validated line range without touching the removed `update` tool

## Risks

### Low

- `\uXXXX` decoding is localized and straightforward
- append/prepend reroute to `insert` is mechanically simple

### Medium

- exposing `calls` publicly may require small prompt/instruction updates if any internal examples still imply CLI-only `useTools`
- deprecated `position` support must not silently permit ambiguous replace behavior

## Rollout Notes

- Treat `position` as deprecated immediately in comments/schema descriptions, even if temporarily supported at runtime.
- Prefer structured `calls` in any new prompt examples that include long markdown bodies.
- Do not widen replace semantics beyond what `contentManager.replace` can validate today.
